## 1. 整體架構圖

```mermaid
flowchart TB
    subgraph Browser["瀏覽器（CODING 前端）"]
        direction TB
        MEM["記憶體<br/>當次暫存、登入憑證<br/>員工／案場清單"]
        SS["分頁暫存 sessionStorage<br/>操作者身分<br/>會計主檔快取"]
        LS["本機長期暫存 localStorage<br/>HUB 員工／案場／班表"]
    end

    subgraph GAS["Google Apps Script 後端"]
        direction TB
        SC["後端快取 ScriptCache<br/>員工權限、案場對照<br/>LINE 暫存"]
        EXEC["單次執行記憶體快取<br/>訂編案號索引"]
    end

    subgraph Sheets["Google 試算表（真實資料）"]
        CHECKIN["打卡系統試算表<br/>員工資料、案場資料"]
        ACCT["會計試算表<br/>廠商、收款帳戶、分類、交易…"]
    end

    Browser -->|"讀寫"| GAS
    GAS -->|"讀取"| Sheets
    Browser -.->|"HUB 父層 iframe 共用資料"| Browser

    SS ---|"關分頁即清"| MEM
    LS ---|"保留 3～7 天"| MEM
```

| 層級 | 典型儲存位置 | 保留多久 | 用途 |
|------|--------------|----------|------|
| 記憶體 | 當次分頁變數 | 分頁開著／LIFF 效期內 | API 去重、登入憑證（**不寫入本機長期暫存**） |
| 分頁暫存 | `tanxin_operator_v1`、會計主檔快取 | 關分頁前／**3 天** | 操作者是誰、bootstrap 主檔 |
| 本機長期暫存 | HUB 員工、案場、班表 | **3～7 天** | HUB 跨模組共用 |
| 後端快取 | 員工權限、案場 map 等 | **6 小時**（平台上限） | 減少試算表讀取 |
| 試算表 | — | 即時 | 唯一真實來源 |

## 2. HUB → 會計身分傳承流程

從官方 LINE 進 HUB，再開會計子頁時，身分怎麼帶過去。

```mermaid
flowchart TD
    A["使用者開官方 LINE"] --> B["HUB 登入<br/>取得 LINE 個人資料"]
    B --> C["得到使用者 ID<br/>顯示名稱、權限等"]
    C --> D["HUB 用 iframe 開會計子頁<br/>網址帶上身分參數"]
    D --> E["網址含 uid、name、permission<br/>及 HUB 的 LIFF 識別"]

    E --> F["會計子頁載入<br/>讀取操作者身分模組"]
    F --> G["從網址寫入分頁暫存<br/>記住操作者是誰"]
    G --> H["啟動會計頁面"]
    H --> I["建立登入工作階段"]

    I --> J{開發模式略過登入?}
    J -->|是| K["用測試帳號<br/>向後端驗證身分"]
    J -->|否| L{在 HUB iframe 內?}
    L -->|是| M["向父層 HUB 要登入憑證<br/>postMessage"]
    M --> N["父層回傳 HUB 的 LIFF 憑證"]
    N --> O["用官方 LIFF 憑證<br/>向後端驗證身分"]
    L -->|否／失敗| P["自己初始化 LIFF<br/>取得憑證後向後端驗證"]

    K --> Q["後端查打卡員工表<br/>確認身分與權限"]
    O --> Q
    P --> Q

    Q --> R["回傳：使用者 ID、權限、顯示名稱"]
    R --> S["用後端結果覆寫<br/>分頁暫存的操作者身分"]
    S --> T["顯示會計畫面"]

    T --> U["站內跳轉時<br/>延續網址參數或分頁暫存<br/>身分不丟失"]
```

**重點**

- 員工唯一鍵：打卡系統「員工資料」的 **官方 LINE 使用者 ID**。
- HUB iframe 會帶 uid、permission、hub_liff_id；子頁若網址參數不見，仍從分頁暫存讀操作者身分。
- **核准權限以後端查表為準**；網址上的 permission 僅供開發略過或過渡顯示。

## 3. 會計頁面開啟流程（現行 vs 目標）

### 3.1 現行路徑（多數頁面）

使用者必須等後端驗完身分，才看得到畫面；主檔也常更晚才載入。

```mermaid
flowchart TD
    START["網頁載入"] --> BOOT["啟動會計頁面"]
    BOOT --> UI["初始化介面與導覽"]
    UI --> INIT["等待建立登入工作階段"]

    INIT --> POLICY["向後端取權限政策<br/>🔴 打 GAS"]
    POLICY --> AUTH["向後端驗證身分<br/>🔴 打 GAS"]
    AUTH --> PERM{權限夠嗎?}
    PERM -->|否| DENY["顯示拒絕訊息"]
    PERM -->|是| APPLY["寫入操作者身分"]
    APPLY --> SHOW["顯示會計畫面"]

    SHOW --> READY["頁面就緒回呼"]
    READY --> NEED{這頁需要主檔?<br/>廠商、收款帳戶等}
    NEED -->|是| CACHE{"分頁暫存有主檔快取<br/>且未過期?"}
    CACHE -->|有| USE["直接用快取<br/>⚪ 不打 GAS"]
    CACHE -->|無| BOOTAPI["向後端載入主檔<br/>🔴 打 GAS"]
    BOOTAPI --> WRITE["寫入分頁暫存"]
    NEED -->|否| DONE["頁面就緒"]
    USE --> DONE
    WRITE --> DONE

    style POLICY fill:#fee,stroke:#c00
    style AUTH fill:#fee,stroke:#c00
    style BOOTAPI fill:#fee,stroke:#c00
    style USE fill:#efe,stroke:#080
```

**現況**：使用者必須等 **登入工作階段（政策 + 身分驗證）** 完成才看到畫面；主檔 bootstrap 多在頁面就緒 **之後** 才載入，需要廠商／收款帳戶列表的頁面會更慢。

### 3.2 目標路徑（有快取就先顯示）

```mermaid
flowchart TD
    START2["網頁載入"] --> OP["從分頁暫存或網址<br/>讀取操作者身分<br/>⚪ 本地"]
    OP --> BCHECK{"主檔快取有效?"}
    BCHECK -->|有| RENDER["立刻顯示主檔相關 UI<br/>⚪ 不打 GAS"]
    BCHECK -->|無| SPIN["顯示 loading / 骨架畫面"]

    RENDER --> BG["背景驗證身分<br/>🔴 打 GAS"]
    SPIN --> INIT2["驗證身分 + 載入主檔<br/>🔴 打 GAS"]
    INIT2 --> RENDER2["顯示畫面"]

    BG --> AUTHOK{驗證成功?}
    AUTHOK -->|否| CLEAR["清除身分<br/>導回登入"]
    AUTHOK -->|是| SYNC["同步操作者身分"]

    RENDER --> NOREFRESH["已有主檔快取<br/>❌ 不背景重讀<br/>（SPEC 18 定案）"]
    RENDER2 --> DONE2["頁面就緒"]

    style RENDER fill:#efe,stroke:#080
    style NOREFRESH fill:#efe,stroke:#080
    style BG fill:#ffe,stroke:#aa0
    style INIT2 fill:#fee,stroke:#c00
```

**目標差異**

| 項目 | 現行 | 目標 |
|------|------|------|
| 首次可互動 | 等身分驗證完成 | 有主檔快取 → **先顯示** |
| 身分驗證 | 擋住畫面 | 背景執行；失敗再清 session |
| 主檔載入 | 頁面就緒後才載 | 進頁先查快取；沒有才向後端要 |
| 背景重讀主檔 | 部分頁會背景預載 | **不做**（靠 CRUD 後清除保證正確） |

## 4. 主檔快取讀寫流程

對應 SPEC 18 §6.1。廠商、收款帳戶、分類等「主檔」怎麼快取、什麼時候打後端。

```mermaid
flowchart TD
    ENTRY["頁面需要主檔<br/>載入或讀取快取"] --> FORCE{強制重抓?}
    FORCE -->|否| READ["讀分頁暫存與記憶體"]
    READ --> FRESH{未超過 3 天?}
    FRESH -->|是| HIT["回傳快取內容<br/>⚪ 不打 GAS"]
    FRESH -->|否| MISS["視同無快取"]

    FORCE -->|是| MISS
    MISS --> INFLIGHT{已有進行中的請求?}
    INFLIGHT -->|是| WAIT["等待同一筆請求完成"]
    INFLIGHT -->|否| API["向後端載入主檔包<br/>🔴 打 GAS"]

    API --> OK{成功?}
    OK -->|否| ERR["顯示載入失敗"]
    OK -->|是| WRITE["寫入分頁暫存與記憶體"]
    WRITE --> RET["回傳主檔內容"]
    WAIT --> RET
    HIT --> DONE["頁面使用主檔資料"]
    RET --> DONE

    style HIT fill:#efe,stroke:#080
    style API fill:#fee,stroke:#c00
```

**主檔內容**：廠商、廠商 LINE 綁定、收款帳戶、分類、訂編案號對照、帳本狀態、列舉值。  
**快取 key**：依操作者使用者 ID 分開存（`tanxin_accounting_bootstrap_v1:{userId}`）。

## 5. 新增／修改／刪除後快取失效流程

使用者改動主檔資料後，快取怎麼清、什麼時候重抓。

```mermaid
flowchart TD
    USER["使用者新增／修改／刪除<br/>廠商、收款帳戶、分類<br/>訂編案號、廠商 LINE 綁定"] --> API["送出 CRUD 請求<br/>🔴 打 GAS"]
    API --> GASOK{後端成功?}
    GASOK -->|否| FAIL["顯示錯誤<br/>快取不動"]
    GASOK -->|是| INV["CRUD 成功後<br/>清除主檔快取"]
    INV --> CLEAR["移除分頁暫存與記憶體"]

    CLEAR --> SAME{同頁還要最新列表?}
    SAME -->|是| RELOAD["強制重抓主檔<br/>🔴 打 GAS"]
    SAME -->|否| NEXT["換頁後第一次載入<br/>自動重抓"]

    API2["廠商資料夾／照片上傳成功"] --> INV

    GASCRUD["後端訂編案號 CRUD 成功"] --> MEMCLR["清除後端單次執行<br/>內的索引快取"]

    style API fill:#fee,stroke:#c00
    style RELOAD fill:#fee,stroke:#c00
    style CLEAR fill:#ffe,stroke:#aa0
```

**原則**：凡會改到 bootstrap 主檔的資料，CRUD 成功後一律 **整包清除** 快取（不只改局部欄位）。  
**不清 bootstrap 快取**：帳本狀態、廠商付款、毛利等交易資料（各頁用即時 API 查）。

## 6. HUB 員工／案場「先顯示舊資料、背景更新」流程

會計表單透過 HUB 上下文讀員工、案場；HUB 端採 **先顯示快取、背景再更新**（與主檔 bootstrap 策略不同）。

```mermaid
flowchart TD
    HUBSTART["HUB 啟動"] --> LOAD["讀本機長期暫存<br/>員工、案場清單<br/>⚪ localStorage"]
    LOAD --> HAS{有快取?}
    HAS -->|是| INSTANT["立刻顯示 HUB 畫面<br/>員工／案場清單"]
    HAS -->|否| WAITUI["顯示 loading"]

    INSTANT --> LIFF["LINE 登入初始化"]
    WAITUI --> LIFF

    LIFF --> FETCH["向打卡後端抓<br/>考勤與案場資料<br/>🔴 打 GAS"]

    FETCH --> DIFF{資料有變?}
    DIFF -->|是| SAVE["寫回本機暫存 3 天<br/>更新畫面與父層變數"]
    DIFF -->|否| SKIP["不觸發多餘重繪"]

    SAVE --> IFRAME["iframe 內的會計子頁"]
    SKIP --> IFRAME

    IFRAME --> CTX["會計讀 HUB 上下文<br/>優先父層清單<br/>其次本機暫存<br/>⚪ 不打 GAS"]
    CTX --> FALLBACK{仍無資料?}
    FALLBACK -->|是| FORMCTX["向後端補抓表單用資料<br/>🔴 打 GAS"]

    style INSTANT fill:#efe,stroke:#080
    style CTX fill:#efe,stroke:#080
    style FETCH fill:#ffe,stroke:#aa0
    style FORMCTX fill:#fee,stroke:#c00
```

**班表**（依年月分 key，TTL **7 天**）同樣先顯示快取，背景再更新當月班表。

## 7. 哪些操作「必須」打後端 vs 可快取

| 操作 | 可快取？ | 儲存／保留多久 | 備註 |
|------|----------|----------------|------|
| **登入驗證** | ❌ 每次進頁需驗 | 後端 6 小時 | 前端仍應呼叫；後端快取減少試算表讀取 |
| **權限政策** | ⚠️ 可短暫記憶體 | — | 現行每次建立工作階段都會打 |
| **主檔 bootstrap** | ✅ | 分頁暫存 3 天 | 有快取且未 CRUD 清除 → 不打 API |
| **HUB 員工／案場** | ✅ 先顯示再更新 | 本機暫存 3 天 | iframe 內讀父層或本機暫存 |
| **HUB 當月班表** | ✅ 先顯示再更新 | 本機暫存 7 天 | 會計較少直接用 |
| **操作者身分** | ✅ 分頁內 | 關分頁前 | 網址參數為初次寫入來源 |
| **CRUD 寫入** | ❌ | — | 成功後清除主檔快取 |
| **收支／請款／毛利列表** | ❌ | — | 交易資料即時查 |
| **考勤／打卡當日** | ❌ | — | 高頻變動 |
| **薪資／個人出勤** | ❌ 或不進本機長期暫存 | — | 敏感資料 |
| **LIFF 登入憑證** | 僅記憶體 | LIFF 效期內 | **不寫本機長期暫存** |

## 8. 為何沒用好快取會覺得慢

1. **身分驗證擋住畫面**：多數頁面在顯示 UI 前，必須完成「權限政策 + 身分驗證」兩次後端往返（冷啟動常 **2～5 秒**）。
2. **主檔載入太晚**：廠商、收款帳戶等常在頁面就緒後才載；若分頁暫存沒快取，使用者已看到外殼仍要等後端讀多張試算表（最長約 120 秒）。
3. **重複載入主檔**：若進頁沒先查快取、或 CRUD 清除後沒強制重抓，可能每次進頁都打 API。
4. **沒讀 HUB 父層快取**：收支登錄若不走 HUB 上下文，會多打一次表單補抓 API。
5. **主檔不做背景重讀是刻意的**：正確性靠 CRUD 後清除；不應每次進頁背景重抓（那會讓「有快取仍慢」）。

**改善方向（對照 §3.2）**：有主檔快取時先顯示畫面；身分驗證改背景執行；進頁先查快取，命中就略過 API；HUB iframe 內優先讀父層員工／案場清單。

| HUB 寫回本機快取 | `saveCache(..., 3 天)` |

| 後端員工權限快取 | ScriptCache `acct_emp_{userId}` |
| 後端案場 map 快取 | ScriptCache `margin_site_map_v1` |

| 操作者身分分頁暫存 key | `tanxin_operator_v1` |
| 主檔快取分頁暫存 key | `tanxin_accounting_bootstrap_v1:{userId}` |
| HUB 員工本機暫存 key | `spa_hub_employees` |
| HUB 案場本機暫存 key | `spa_hub_projects` |
| HUB 班表本機暫存 key | `spa_hub_schedule_{YYYY}_{MM}` |
