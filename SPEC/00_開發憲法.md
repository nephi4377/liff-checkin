# 開發憲法（CODING 大改／新功能清單）

> **適用時機**：新模組、跨多檔、改 API 契約或行為不明時。小修可略過本檔。  
> **共通約束**（LOG、`@STABLE`、git、部署、欄位字典等）一律以 `.cursor/rules/` 為準，本檔不重複。

---

## 1. 先對齊範圍

1. 確認要動的是 **哪個模組**（`modules/`、`spa/`、`shared/` 等）。
2. 讀該模組 **`SPEC/*.md`** 當預期；與現行程式不符時**以程式為準**，更新 SPEC（程式明顯是 bug 才改程式）。
3. 用搜尋確認是否**已有函式／API** 可沿用，避免平行實作兩套。

使用者若只要簡答或單點修改，不必拉完整流程。

---

## 2. 實作節奏

- **一次一個可驗證目標**（例如：先修 API 回傳，再改 UI）。
- 改動集中、diff 小；不順手重構無關檔案。
- 不確定時先問使用者，或先讀碼回報影響範圍再動手。

---

## 3. 驗證

| 類型 | 做法 |
|------|------|
| 純邏輯 | 跑對應單元測試（若有） |
| 網頁／互動 | 起靜態站 + 手動或 Playwright（指令見 skill **`cloud-agent-runbook`**） |
| 接 GAS | 在 **`backend/`** 倉庫改；前端只改呼叫與設定 |

能跑就跑；測不動則說明手動驗證步驟。

---

## 4. 可選：Cursor 子代理

僅在**範圍大、需並行查多處**時使用 Task（`explore` 查碼、`shell` 跑指令等）。  
單人日常開發**不強制**多代理人編排。舊版分工 SOP 見 [`SPEC/archive/多代理人協作_參考.md`](archive/多代理人協作_參考.md)。

---

## 5. 收工

- 大改完成：依 rule `00-general-plain-language` 寫 **LOG**；相關 **SPEC** 與程式同步。
- 要上 GitHub：依 rule **`CODING-upload-deploy`** 執行 `upload.bat`。
