# 添心生產力助手與企業自動化生態系——全專案系統大綱 (PROJECT_MAP)
> **版本：v2.1 (2026-06-01) | 適用對象：設計總監、系統營運與維護 AI 代理人**

本數位藍圖專為「添心設計」的企業自動化與高階管理量身打造。本系統採用 **Micro-Service at Edge + Cloud GAS (Google Apps Script)** 的混合雲端架構，整合了邊緣端行為分析、雲端考勤與排班管理、行動端現場日報提報、自動化備份守護、財務記帳及大師級投資決策心智。

以下為本生態系的完整結構、營運生活動線與核心安全規範。

### ❶ 添心生產力助手 (Electron Desktop 客戶端)
*   **空間角色**：邊緣端（Edge）行為分析與打卡的前瞻性桌面門面。
*   **機能定義**：常駐於員工電腦背景，執行非侵入式的視窗標題監測與今日運行時數累算。不錄音、不錄影、不截圖、不側錄按鍵（嚴守隱私紅線）。提供小秘書 Mascot 本地影像快取、iCloud 行事曆同步以及打卡與通知彈窗提示，是同仁每日工作的第一站。

### ❷ backend (GAS 雲端後端)
*   **空間角色**：企業雲端業務與資料持久化的邏輯核心（Brain on Cloud）。
*   **機能定義**：以 Google Apps Script (GAS) 部署，並將 Google Sheets 試算表作為「分散式關聯資料庫」。
    *   `CheckinSystem` 負責考勤、排班表、休假與加班的審計快取與定時通知。
    *   `project-console` 負責案場資料、施工日誌的寫入與 Firebase 即時橋接。
    *   `core_library` 作為公共依賴庫，封裝了對 Gemini AI、Dropbox 雲端相簿與 LINE Messaging API 的通訊邏輯。

### ❸ CODING (Web 戰情前端)
*   **空間角色**：設計總監與高階主管專用的「日報與出勤戰情室」 (Executive Dashboard)。
*   **機能定義**：採用極致精細的 Tailwind UI 網頁設計，不經由後端進行高成本的聯表查詢（JOIN），而是直接透過瀏覽器發送 `Promise.all` 聯立後端考勤與日誌資料。提供「人員模式（縱向時間軸）」與「專案模式（橫向案場圖片）」雙視角，具備全域檢索與缺交日報紅字警示。

### ❹ 行動端 LIFF / LINE 綁定
*   **空間角色**：第一線工務同仁的行動回報端與身分識別卡。
*   **機能定義**：嵌入於 LINE 官方帳號的 LIFF 網頁，負責員工的 LINE 身份綁定。工務同仁可在現場進行 GPS 定位打卡、請假提報，並直接透過手機拍照上傳當日案場進度。

### ❺ TanxinTools (企業自動化與財務安全工具)
*   **空間角色**：守護企業資料與帳務健康的守護進程（Data & Financial Guard）。
*   **機能定義**：
    *   **BGM (Backup Guard Monitor v3)**：使用 Python 常駐 Daemon，24 小時不間斷巡檢 Dropbox 及 Google Drive 的備份目錄與關鍵檔案完整性。校驗「Fuse 虛擬保險絲檔案」防範勒索軟體，異常時即時推播並生成圖表。
    *   **Accounting System (本地記帳系統)**：基於 SQLite 與 Web 服務，支持 OCR 自動辨識發票與收據，保障企業資金往來數據的安全留存。

### ❻ 投資分析 (Nephi 投資決策工作區)
*   **空間角色**：基於系統性心智模型的資產管理與決策中心。
*   **機能定義**：提供巴菲特（護城河與內在價值 DCF 計算）、查理蒙格、彼得林區（PEG 成長分析）、葛拉漢（純數字安全邊際）、費雪以及「總經×情緒×地緣（4區間價格預測）」等 6 大大師框架。分析結果儲存為結構化 JSON 並自動呈現在戰情主頁上。

## 🔄 2. 核心營運與生活動線（Data Flow）

### 🅰️ 員工出勤打卡生活動線（Client to Cloud）

此動線貫穿了「本地前端 UI 防禦鎖定 ➔ 本機大腦轉發 ➔ 雲端邏輯處理 ➔ 雲端 Sheets 寫入 ➔ 前端回傳解鎖」的閉環，確保資料不會因使用者連點或連線延遲而重複寫入。

```
 [ Stats Window ] (UI 報表視窗)
   │
   ├─ 1. 同仁點擊「✅ 打卡發送」按鈕
   ├─ 2. 【防禦鎖定】按鈕立即變更為 Disabled (鎖定狀態，防連點/併發)
   │
   ▼ (網頁調用 window.reminderAPI.directCheckin() IPC 橋接)
 [ AppCore.js ] (Electron 客戶端大腦)
   │
   ├─ 3. 接收 `direct-checkin` 訊號，調用 `CheckinService` 通訊模組
   │
   ▼ (發送 HTTPS POST 請求，攜帶 userId, pcName, timestamp)
 [ CheckinSystem / WebApp.gs ] (GAS 後端入口)
   │
   ├─ 4. 路由分發：呼叫 `Logic_Checkin.js` 處理打卡
   ├─ 5. 身分與設備驗證：調用 `EmployeeLogic.js` 判定該電腦是否綁定此 userId
   ├─ 6. 讀取排班與假勤：比對 `ScheduleLogic.js` 今日班表
   ├─ 7. 寫入快取與實體：更新「打卡全域快取」，並寫入 Google Sheets 試算表 `打卡歷史紀錄` 工作表
   │
   ▼ (回傳 JSON 成功響應狀態，並攜帶最新打卡時間)
 [ Stats Window ] (解鎖與對齊)
   │
   ├─ 8. 接收到後端成功訊息，【解鎖】「打卡發送」按鈕
   └─ 9. 自動執行 `refreshStats()` 重新整理統計，出勤時間欄位立即更新
```

### 🅱️ 施工日報上傳與戰情呈現動線（Reporter to Dashboard）

此動線是設計總監掌握全案場進度的生命線。現場照片與回報經過「實體轉移 ➔ AI 優化摘要 ➔ 關聯聯表 ➔ 動態瀑布渲染」後，以極致直覺的資訊流呈現於總監的螢幕上。

```
 [ 現場工務手機端 ] (LINE LIFF 提報)
   │
   ├─ 1. 手機拍下工地现场、輸入施工進度描述、選擇案場
   └─ 2. 送出日報
   
   ▼ (發送 HTTPS API 請求至 project-console/WebApp.gs)
 [ project-console (GAS 雲端後端) ] (資料接收與處理)
   │
   ├─ 3. 【相片物理儲存】：調用 `dropbox_api.js` 將相片傳輸至 Dropbox 指定案場目錄，並生成免登入下載連結
   ├─ 4. 【AI 內容優化】：調用 `core_library.js` 之 Gemini API，依工務提報內容進行潤飾與核心重點摘要
   ├─ 5. 【資料庫寫入】：寫入 Google Sheets 的 `ProjectLog (施工日誌)` 工作表
   ├─ 6. 【LINE 推播】：呼叫 LINE Messaging API 推播通知給相關群組
   
   ▼ (設計總監開啟日報戰情網頁 CODING/daily_report.html)
 [ CODING / daily_report_main.js ] (前端聯表與渲染)
   │
   ├─ 7. 【重度聯集 (Heavy Join)】：利用 `Promise.all` 同時在背景拉取：
   │       * `project-console/WebApp.gs` ➜ 取得今日所有專案與施工日誌
   │       * `CheckinSystem/WebApp.gs` ➜ 取得今日所有人員的出勤與假勤狀態
   │
   ├─ 8. 【前端 Join 處理】：以 `UserID` 為鍵，在瀏覽器本地將「考勤/假勤」與「施工日誌」內聯比對
   │
   ├─ 9. 【時間軸自動補完 (Timeline Auto-Complete)】：
   │       * 自動在人員視角下補足指定區間內的每一天。
   │       * 比對若無日報：
   │         ├── 比對假勤 API ➔ 為特休、事假、補休者：標註假別，並將該日回報卡片【淡化 60%】(降噪)
   │         └── 比對為出勤或班表應上班 ➔ 反紅警告並顯示【⚠️ 缺交報告】
   │
   ├─ 10. 【Waterfall Engine (瀑布渲染)】：
   │       * 使用 JavaScript `reduce` 方法，動態將資料依「人員模式」或「專案模式」進行流暢無感的視角轉換
   │
   └─ 11. 最終呈現：總監看到高清晰、毫秒級載入 (Drive sz=w128)、帶有 AI 摘要與缺交紅字警示的完美戰情室
```

### 🆃 備份守護巡檢動線 (BGM Python Daemon)

此動線全天候保護企業數據免受硬體故障、意外刪除或勒索病毒攻擊：

```
 [ Windows 工作排程 / Daemon 守護進程 ] ➔ 執行 `run_daemon.bat`
   │
   ▼ (每隔 15 分鐘定時啟動核心週期 `run_cycle`)
 [ backup_guard.py ] (BGM 備份守護核心)
   │
   ├─ 1. 連接 Dropbox / Google Drive API。
   ├─ 2. 深度遍歷與掃描所有備份目錄，核對 `BACKUP_DROPBOX_V3_SPEC.md` 定義之規格。
   ├─ 3. 【虛擬保險絲 (Fuse) 校驗】：
   │       * 檢查特定隱藏的 `fuse` 檔案完整性與唯讀防護狀態。
   │       * 比對本地與雲端鏡像的時間戳與雜湊值。
   │
   ├─ 4. 【狀態處置】：
   │       ├── [正常] ➜ 默默寫入 `bgm_runtime_log.py`，保持安靜，更新 runtime 狀態
   │       └── [異常] (如 Fuse 缺失、備份中斷、檔案遭勒索軟體惡意竄改等)：
   │             ├── a. 立即寫入 `alerts.json`
   │             ├── b. 觸發 `bgm_report_charts.py` 與 `bgm_report_tree.py` 生成備份異常樹狀圖與可視化告警網頁
   │             └── c. 觸發推播警報，強制提醒總監與系統維護者處理
```

## 🛠️ 4. 總監級安全施工規範（修改安全指南）

為保障本生態系在多端運作下的極致穩定，所有開發代理（AI Agents）與維護人員必須嚴格遵守以下三層防禦施工守則：

### 🔒 守則一：防禦連點與併發衝突 (Concurrency Guard)
*   **痛點**：Google Apps Script 與 Sheets 的寫入有約 2 到 5 秒的延遲。如果使用者在連線期間重複點擊按鈕，會造成 Google Sheets 產生重複重複的打卡列，損毀資料庫結構。
*   **規範**：
    1.  所有前端 UI（例如 `Stats Window` 中的「✅ 打卡發送」、LINE LIFF 中的提交鍵）在點擊後，**必須第一時間將按鈕狀態設為 `disabled` 並鎖定**。
    2.  唯有在接收到後端回傳的 JSON 成功（或失敗）響應後，才允許解鎖按鈕。
    3.  嚴禁因美觀原因移除此 UI 鎖定防禦機制。

### 🔄 守則二：熱更新全程序重啟機制 (Relaunch Protocol)
*   **痛點**：當小助手進行背景熱更新並下載增量補丁（Patch）後，如果只用 JavaScript 進行頁面或服務的軟重載（Soft-Reload），Electron 的 IPC (Inter-Process Communication) 通訊通道不會被釋放。此時重複註冊事件會觸發 `Attempted to register a second handler` 錯誤，導致程式瞬間閃退崩潰。
*   **規範**：
    1.  熱更新下載補丁完成後，**必須呼叫 `app.relaunch()` 進行整套程序的強制實體重啟**。
    2.  `setupIpcHandlers` 在註冊任何 channel 前，必須先執行安全清理：
        ```javascript
        try { ipcMain.removeHandler(channelName); } catch (e) {}
        ```
    3.  此流程不可簡化，確保 IPC 機制在補丁更新後 100% 潔淨無殘留。

### 🚀 守則三：標準上線部署流程 (One & Only Deployment Path)
*   **痛點**：人工手動執行 `clasp push` 容易遺漏備份，且會造成本地代碼、GitHub 倉庫與雲端 GAS 運行代碼版本不一致（出現「程式碼在某人電腦，雲端不知道是哪版」的災難）。
*   **規範**：
    1.  當對 `backend` 進行任何修改、需要將系統推上線時，**唯一路徑是執行 `backend` 根目錄下的 `upload.bat` 批次檔**。
    2.  `upload.bat` 會嚴格執行以下流程：
        *   **安全備份**：自動調用 Robocopy 將整個專案鏡像備份至本機外層的 `..\BAK\Backend_GAS_時間戳_電腦名\` 目錄下。
        *   **Git 提交**：自動切換至 `main` 分支 ➜ `git add .` ➜ 進行 Commit。
        *   **GitHub 自動編譯**：`git push origin main`。
        *   **GitHub Actions 自動發布**：由雲端 CI/CD Workflow 自動執行 `clasp push` + `clasp deploy`。
    3.  **禁止在本地終端機直接執行 `clasp push`**（除非斷網或緊急偵錯，且必須事後同步 Git）。
