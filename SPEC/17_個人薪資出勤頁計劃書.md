# 個人薪資出勤頁 — 計劃書（草案）

**版本**：v0.4（2026-06-28）  
**狀態**：Phase A0／A1 已上線；**Phase B1 薪資核對已部署**（2026-06-28）；B2 實領歷史待會計串接  
**關聯**：`15_會計系統模組規格書.md` Phase 2e 薪資、`CheckinSystem` 打卡／假勤

---

## 1. 這是什麼？

給**每一位員工**用的「個人專區」：登入後只看**自己的**出勤、假別、薪資，不用進會計後台，也看不到別人的資料。

開發期先限**權限 ≥ 5**（主管／老闆）試用；確認流程沒問題後，再開放給**一般在職員工**（仍只能看本人）。

---

## 2. 為什麼要做？


| 現況                       | 問題                            |
| ------------------------ | ----------------------------- |
| 薪資在試算表，只有權限 ≥ 4 的財務／主管能看 | 員工不知道自己當月怎麼算、有疑問只能口頭問         |
| 出勤、請假在 Checkin 系統        | 和薪資分開，對帳時容易漏看                 |
| 獎金、補卡、假勤異常               | 多半 LINE 或口頭，缺少「每個發薪週期一次交齊」的入口 |


**目標**：一個頁面把「我這個月上了幾天班、請了什麼假、薪資怎麼來、有沒有要補說明的」集中在一起，並留下紀錄。

---

## 3. 誰能用？


| 階段       | 權限           | 說明                             |
| -------- | ------------ | ------------------------------ |
| **開發試用** | ≥ 5          | 入口先隱藏或灰階，僅內部測流程                |
| **正式上線** | 在職員工（建議 ≥ 1） | 後端強制「只能查自己的 user_id」，不可改參數偷看他人 |
| **管理端**  | ≥ 4（維持現狀）    | 會計後台仍可看全員薪資、審核員工送出的說明          |

## 4. 功能項目一覽

下面每一項都是計劃要有的能力；括號內是白話說明。

### 4.1 個人總覽（首屏）

- **本月摘要卡片** — 出勤天數、請假時數、預估／已核薪資區間（例：115/06）
- **待辦提醒** — 尚有未補卡、假勤待審、薪資審核未送出、主管退回需修改
- **快速入口** — 出勤明細、假別紀錄、薪資明細、本週期審核送出



### 4.2 出勤狀況（唯讀 + 申訴／補說明）

- **月曆／列表** — 每日上下班時間、案場、遲到／早退標記（資料來源：Checkin `打卡紀錄`）
- **異常標示** — 缺卡、僅單邊卡、與排班不符（對照 `排班歷史紀錄`）
- **出勤申訴（Phase A1）** — 逐日明細旁「提出說明」；可附照片／截圖；**15 日內**可送；主管審核通過後可**手動補登**修正紀錄（見 §4.7）
- **匯出本人 PDF／列印** — 選配；優先 Phase 2 再做

#### 4.2.1 技術對接（與管理版共用後端）

| 項目 | 說明 |
| --- | --- |
| 頁面 | `attendance_report.html?mode=personal`（路由 `#/my-personal`） |
| 出勤 API | `get_report`（`userId` 鎖定本人） |
| 假勤 API | `get_my_leave_requests` |
| 遲早退算法 | **後端** `CheckinSystem/Logic_AttendanceReport.js` → `_computeReportLateEarly_` |
| 規格詳述 | `backend/CheckinSystem/SPEC/出勤儀表板與報表規格分析V2.md` §2.5、§2.5.6 |

> **白話**：個人頁的遲到／早退分鐘與管理員出勤儀表板**同一套公式**；後端修正後個人頁自動同步，前端不重算。  
> **彈性早到範例**（表定 08:30、17:30、彈性 30 分）：07:53 到 → 應 17:00 可下班；17:10 走 → 早退 **0** 分。

> 內規提醒：未打卡應當天或隔天補註明原因；不得代打卡（見設計師規範 V7.0）。系統申訴時效為**異常日起 15 日內**（逾期仍可送，標記「逾期申訴」）。



### 4.3 假別紀錄

- **已核准／待審／退回** — 特休、事假、病假、排休等（來源：Checkin `排班歷史紀錄`）
- **剩餘特休** — 依 §4.8 年度統計工作頁計算（已用天數＋人事給額）；無給額時僅顯示「已用 X 天」
- **申請紀錄連結** — 可跳轉既有 Checkin 請假流程（個人頁不重做完整假單 UI 也可）

> 內規提醒：特休原則 3 個工作天前申請；超過 3 天連假需交接。



### 4.4 薪資查看（本人）

- **依發薪週期選擇** — 例 `115/05`、`115/06`（來源：會計 `PAYROLL_SPREADSHEET_ID` 薪資分頁）
- **明細列** — 本薪、加班、扣款、實發、備註（欄位對齊 `MasterSchema` payroll entity）
- **狀態** — 草稿／已發放／員工已確認／有異議
- **附件** — 若該期有薪資單 PDF 或截圖，可顯示連結（選配）

**重要**：後端 API 必須帶 `employee_user_id = 登入者`，禁止 list 全表。

### 4.5 每週期「薪資審核」送出（員工 → 主管）

每個發薪週期結束後（或公告的截止日前），員工在個人頁一次完成：


| 區塊       | 員工作什麼               | 主管作什麼             |
| -------- | ------------------- | ----------------- |
| **薪資確認** | 勾選「已核對無誤」或勾選有疑問並填說明 | 查看、回覆、必要時修正試算表後重送 |
| **獎金申請** | 填項目（案號、事由、金額、附件）可多筆 | 核准／駁回；核准者寫入薪資或另表  |
| **假勤審核** | 對該週期內異常假勤逐筆確認或補說明   | 對照 Checkin 核准紀錄   |


- **截止時間** — 由主管在後台或試算表設定（例：發薪日前 3 天）
- **送出後鎖定** — 員工不可改，除非主管「退回修改」
- **稽核** — 每次送出、核准、退回寫入 `AUDIT_SPREADSHEET_ID`



### 4.8 個人假勤年度統計（Phase A2）

**目的**：員工在個人頁查看**一整年**（曆年或特休週年，見 §4.8.3）各假別用量與出勤摘要；後端**不每次**從頭掃 `打卡紀錄`＋`排班歷史紀錄`，改讀**獨立工作頁**快取。

#### 4.8.1 畫面（個人頁新分頁「年度統計」）

- **年度選擇** — 預設當年度；可切換前 1～2 年（唯讀）
- **摘要列** — 特休／病假／事假／補休／加班（天）、實際出勤、缺勤、遲到／早退（分）
- **特休區塊** — **已用天數**（小數 1 位，例 7.5 天）；若人事有給額則顯示 **剩餘 = 給額 − 已用**
- **明細（選配）** — 按月折疊：各月特休／病假／事假小計（不必列出逐日，逐日仍走 §4.2／4.3）
- **資料新鮮度** — 顯示工作頁 `lastUpdatedAt`（例：「統計更新至 2026-06-28 06:30」）；超過 24 小時可標「建議重新整理」

#### 4.8.2 獨立工作頁 `假勤年度統計`（Checkin 試算表新分頁）

每次**增量或全量重算**後寫入此表；個人頁 API **優先讀表**，僅在缺列或強制刷新時重算。

| 欄位 | 說明 |
| --- | --- |
| statYear | 統計年度（例 `2026`） |
| employeeUserId / employeeName | 員工 |
| employeeGroup | 組別 |
| annualLeaveDays | 特休已用（天，可小數） |
| sickLeaveDays | 病假（含生理假） |
| personalLeaveDays | 事假 |
| compensatoryDays | 補休 |
| overtimeDays | 加班（天） |
| checkInDays | 實際出勤天 |
| absentDays | 缺勤天 |
| lateMinutes / earlyMinutes | 遲到／早退分鐘 |
| scheduledRestDays | 應休（排班／標準制母數內） |
| annualLeaveQuota | 年度特休給額（天）；無則空白 |
| annualLeaveRemaining | 剩餘特休（有給額時才算） |
| monthlyJson | 各月小計 JSON（選填，減少列數） |
| **lastUpdatedAt** | **最後重算時間**（ISO 或試算表日期時間） |
| sourceRangeStart / sourceRangeEnd | 本次納入的日期區間 |
| recomputeTrigger | 觸發來源：`nightly`／`leave_approved`／`manual`／`on_demand` |

**複合鍵**：`statYear` + `employeeUserId`（每人每年一列）。

#### 4.8.3 特休能算出幾天嗎？— **可以**

沿用 Checkin `Logic_AttendanceReport` 既有算法（與薪資核對 B1 的 `stats.annualLeave` 同源）：

| 情境 | 計入天數 |
| --- | --- |
| 全天特休、無打卡 | **1 天** |
| 時段特休（例 `特休[08:00-12:00]`） | 時數 ÷ **8**（例 4h → **0.5 天**） |
| 同日多段特休 | 各段相加 |
| 國定假日上的核准特休 | **計入**（與出勤報表一致） |
| 待審假單 | **不計**；核准後觸發該員該年重算 |
| 小數顯示 | 前端顯示 **1 位小數**；試算表可存完整精度 |

**剩餘特休** = `annualLeaveQuota`（人事維護）− `annualLeaveDays`（系統加總）。無給額欄位時個人頁只顯示「今年已休 X 天」，不顯示剩餘。

**統計年度邊界（已決）**：

- **個人頁年度統計**：採 **曆年制**，固定 1/1～12/31
- **離職結算**：另走 **週年制**，依到職日或人事表「特休起算日」另算，不混入個人頁年度統計

#### 4.8.4 何時重算（避免全表掃描）

| 時機 | 作法 |
| --- | --- |
| **每日排程**（建議 06:00） | 當年度、在職員工全量或「昨日有異動」增量 |
| **假勤核准／銷假** | 事件觸發：只重算該 `employeeUserId` + 當 `statYear` |
| **補登修正打卡** | 同上（若影響缺勤／出勤天數） |
| **個人頁「重新整理統計」** | `on_demand`：該員當年一列；限流（例 5 分鐘 1 次） |
| **管理端** | perm ≥ 4 可強制全員重算（寫入 `recomputeTrigger=manual`） |

重算時仍呼叫 `_getAttendanceDataForReport_(yearStart, yearEnd, userId)` **一次**，結果寫入工作頁；之後查詢只讀一列。

#### 4.8.5 API（草案）

**單一 action**：`leave_year_stats`（比照 `attendance_appeal` 集中一支）

| `mode` | 誰 | 說明 |
| --- | --- | --- |
| `mine` | 員工 | `operatorId` 鎖本人；`year` 選填，預設當年 |
| `refresh` | 員工 | 觸發本人當年重算後回傳 |
| `list` | perm ≥ 4 | 全員某年列表（管理儀表板用，選配） |

GET only；寫入工作頁在後端重算函式內完成，不另開 POST action。

### 4.7 出勤申訴（Phase A1）— 已決策

**目的**：員工對系統判定的遲到、缺卡、離場未填等提出說明；**審核通過後由主管執行手動補登**（不自動改打卡）。

#### 4.7.1 決策摘要（2026-06-27）


| 項目   | 決策                                                                          |
| ---- | --------------------------------------------------------------------------- |
| 修正方式 | **B**：審核通過 → 主管在出勤儀表板**手動補登**（留審核人、申訴編號）                                    |
| 審核者  | **權限 > 3**（即 perm ≥ 4，與假勤審核、出勤儀表板同一條件）                                      |
| 時效   | **異常日起 15 日內**為一般申訴；超過仍可送，狀態加標 **逾期申訴**                                     |
| 附件   | **要** — 照片、截圖（沿用假勤／施工回報類 Drive 上傳模式）                                        |
| API  | **不新增多支 action** — 只加 **一支** `attendance_appeal`，用 `mode`／`op` 分流（見 §4.7.6） |




#### 4.7.2 員工端（`#/my-personal` 出勤分頁）

- 有異常標記的日期列顯示 **「提出說明」**
- 表單：日期（鎖定）、異常類型（自動帶入或選）、原因（**必填**，有內容即可；不設最低字數，是否充分由主管審核）、**快速標籤**（依異常類型帶入常用說明，可複選後再改）、附件（選填，可多檔）
- 同一員工、同一日期、同一異常類型：**僅一筆「待審」**（避免重複送）
- 送出後顯示狀態：待審／已備註／已修正／已駁回／逾期申訴
- **已修正**：表示主管已補登並結案



#### 4.7.3 主管端

- 入口：假勤審核儀表板新增分頁 **「出勤申訴」**（不另開新路由亦可）
- 列表：**perm > 3** 可看全員待審（與假勤審核同一批人）
- 每筆可：**核准（需補登）**／**僅備註不修正**／**駁回**（必填回覆）
- **核准（需補登）**：開啟既有補登流程（帶入申訴的日期、建議上下班時間）；補登成功後申訴狀態 → **已修正**，寫入 `manualCheckinRef`
- 所有操作寫入稽核（誰、何時、原狀態→新狀態）



#### 4.7.4 狀態機

```
待審 ──核准(需補登)──▶ 待補登 ──主管補登成功──▶ 已修正
  │                      │
  ├──僅備註──────────────▶ 已備註
  └──駁回──────────────▶ 已駁回（員工可改內容再送一筆新編號）
```

#### 4.7.6 API — 集中一支（Checkin `attendance_api`）

**原則**：比照 `get_my_leave_requests`（`mode=self|management`），**只新增一個 action 名稱**，避免 WebApp 白名單與 handler 一直膨脹。

##### 唯一 action：`attendance_appeal`

後端實作：`_handleAttendanceAppeal_(params, method)` 一個入口，內部分流。

**GET**（查詢）


| `mode`    | 誰            | 回傳                                      |
| --------- | ------------ | --------------------------------------- |
| `mine`    | 任何在職員工       | 本人申訴列表（`operatorId` 鎖本人）                |
| `pending` | **perm > 3** | 待審／待補登列表（全員，不分組別）                       |
| `status`  | 員工           | 可選：帶 `targetDate` 查該日是否已有申訴（個人頁列上顯示狀態用） |


共用參數：`operatorId`（必填）、`page=attendance_api`、`action=attendance_appeal`

**POST**（寫入，JSON body）


| `op`      | 誰            | 說明                                                                        |
| --------- | ------------ | ------------------------------------------------------------------------- |
| `submit`  | 員工           | 新增申訴；`operatorId`＝本人；含 `reason`、附件（Base64 或 Drive URL）                    |
| `resolve` | **perm > 3** | `appealId` + `decision`（`approve_fix`／`note_only`／`reject`）+ `reviewNote` |


附件上傳邏輯**內嵌在** `_handleAttendanceAppeal_`，複用假勤 `_handleLeaveRequest_` 的 Drive 上傳，不另開 action。

##### 與既有 API 的關係


| 既有                      | A1 不重複做          |
| ----------------------- | ---------------- |
| `get_report`            | 仍負責出勤明細；申訴不改報表算法 |
| `add_manual_checkin`    | 審核通過後仍呼叫此支補登     |
| `get_my_leave_requests` | 假勤走假勤；申訴只做出勤異常   |




##### WebApp 註冊（僅 +1 行）

```text
'attendance_appeal': () => _handleAttendanceAppeal_(params, 'GET')   // doGet
// doPost 路由內：action === 'attendance_appeal' → 同上，method POST
```



#### 4.7.7 驗收

- [ ] 員工只能送、只能看自己的申訴
- [ ] 15 日內／逾期標記正確
- [ ] 附件可上傳、主管可開啟
- [ ] 審核通過 + 補登後，報表該日紀錄更新、申訴為「已修正」
- [ ] perm ≤ 3 無法審核、無法代送

#### Phase B1 — 期別選擇（2026-06-28 決策）

個人頁「薪資核對」的**結算期別**下拉，需能切換多期，不能只顯示單一月份。


| 薪資類型           | 下拉內容（由新到舊）                                  | 預設選中                                    |
| -------------- | ------------------------------------------- | --------------------------------------- |
| **月薪-10**      | **本月（進行中）**、**上月**、**上上月** — **共 3 期**（含本期） | **最近已結束的完整曆月**（例：6/28 預設 5 月，因 6 月尚未結束） |
| **日薪-1/11/21** | **最近一期**、**上一期**、**上上期** — **共 3 期**        | 最近已結束的一旬                                |
| **日薪-1/16**    | 同上 — **共 3 期**                              | 最近已結束的半個月                               |


每筆選項顯示：`相對標籤 · 起迄日期 · 發薪日`（後端欄位 `displayLabel`、`submitHint`）。

- 送審時間**不限制**；員工可補送較早期別（若該期尚無待審中申請）。
- 日薪員工**不試算勞健保**（個人自行處理）。

#### Phase B1 — 勞健保與員工設定（2026-06-28 決策）

| 項目 | 決策 |
|------|------|
| 工會 | **不納入系統**；個人自行處理，UI 不提供工會欄位 |
| 日薪 | **一律不加保**；後端強制不扣款，員工編輯頁隱藏勞健保區 |
| 月薪勞健保 | 依政府投保級距表 + 固定費率試算個人負擔（115/01：勞保 12.5%×20%；健保 5.17%×30%×（1＋眷屬人數）） |
| 投保級距 | 員工資料試算表欄位「投保級距」；編輯頁**下拉選**政府級距表；**預設對齊底薪** |
| 健保眷屬 | 員工資料欄位「健保眷屬人數」（0～10，不含本人）；每位眷屬同級距另計健保自付 30%；勞保僅本人 |
| 資料來源 | 員工資料表中文欄位（薪資類型、發薪規則、底薪、全勤、津貼、保險類型、投保級距、健保眷屬人數） |
| 試算免責 | 「預估僅供核對，以主管審核通過後為準。」 |

**後端**：`Logic_InsurancePremium.js`（試算）、`Logic_PayrollReview.js`（`payroll_review` action）、`EmployeeLogic.js`（欄位補全）。  
**前端**：`payroll_review_panel.js`、`employee_editor.html`；主管審核 `approval_dashboard.html` 分頁。

#### Phase B1 — API 摘要

`action=payroll_review`（Checkin `attendance_api`）

| mode / op | 誰 | 說明 |
|-----------|-----|------|
| GET `context` | 員工 | 薪資設定、期別列表、出勤快照、保險試算、既有申請 |
| GET `mine` | 員工 | 本人申請列表 |
| GET `pending` | perm ≥ 5 | 待審列表 |
| POST `submit` | 員工 | 送出薪資核對（含加班、遠程津貼自填） |
| POST `resolve` | perm ≥ 5 | 核准／退回（主管填獎金） |

試算表分頁：`薪資核對申請`。


**時機**：會計薪資試算表／發放紀錄與 Checkin 個人頁 API **完全串接後**再實作 UI；本階段僅鎖規格。

**目的**：員工在個人頁除「當期試算核對」外，可查看**已發放**的歷史實領，作為對帳與存查依據（非試算、以主管／會計確認後金額為準）。


| 項目      | 規格                                                                                  |
| ------- | ----------------------------------------------------------------------------------- |
| 顯示範圍    | 最近 **12 個月**（依發薪日或 period 倒序）                                                       |
| 資料來源    | 會計 `PAYROLL_SPREADSHEET_ID` 薪資分頁（或 `payroll` entity）；`employee_user_id` 鎖本人         |
| 每列欄位    | 期別（如 115/05）、發薪日、本薪、津貼、加班、扣款（勞健保等）、獎金、**實發金額**、狀態（已發放／已確認）                          |
| 與 B1 關係 | B1「薪資核對申請」= 送審前試算；B2 = 發放後唯讀歷史；兩者不可混用同一列                                            |
| API（草案） | `payroll_review` `mode=history` 或會計 `personal_payroll_history`；GET only；`months=12` |
| 權限      | 本人；perm ≥ 4 管理端另途徑                                                                  |
| 資安      | 不 cache 至 localStorage；稽核 `view_own_payroll_history`                                |
| 驗收      | 僅能看到自己；缺資料期顯示「尚無發放紀錄」；金額與試算表一致                                                      |
