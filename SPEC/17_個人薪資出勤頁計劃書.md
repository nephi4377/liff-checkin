# 個人薪資出勤頁 — 計劃書（草案）

**版本**：v0.6（2026-07-23）  
**狀態**：Phase A0／A1 已上線；**Phase B1 薪資核對已部署**（2026-06-28）；**B1.1 薪資審核／匯款已移至會計**（2026-07-01）；**B2 實領歷史 MVP**（2026-07-04，發薪後明細＋LINE 通知）；**A2 假勤年度統計 Slice 2～4 MVP**（2026-07-23，待部署驗收）  
**關聯**：`15_會計系統模組規格書.md` Phase 2e 薪資、`CheckinSystem` 打卡／假勤

---

## 1. 這是什麼？

給**每一位員工**用的「個人專區」：登入後只看**自己的**出勤、假別、薪資，不用進會計後台，也看不到別人的資料。

開發期先限**權限 ≥ 5**（主管／老闆）試用；確認流程沒問題後，再開放給**一般在職員工**（仍只能看本人）。

---

## 2. 為什麼要做？


| 現況                       | 問題                            |
| ------------------------ | ----------------------------- |
| 薪資在試算表，只有權限 ≥ 4 的財務／主管能看 | 員工不知道自己當月怎麼算、有疑問只能口頭問         |
| 出勤、請假在 Checkin 系統        | 和薪資分開，對帳時容易漏看                 |
| 獎金、補卡、假勤異常               | 多半 LINE 或口頭，缺少「每個發薪週期一次交齊」的入口 |


**目標**：一個頁面把「我這個月上了幾天班、請了什麼假、薪資怎麼來、有沒有要補說明的」集中在一起，並留下紀錄。

---

## 3. 誰能用？


| 階段       | 權限           | 說明                             |
| -------- | ------------ | ------------------------------ |
| **開發試用** | ≥ 5          | 入口先隱藏或灰階，僅內部測流程                |
| **正式上線** | 在職員工（建議 ≥ 1） | 後端強制「只能查自己的 user_id」，不可改參數偷看他人 |
| **管理端**  | ≥ 4（維持現狀）    | 會計「薪資待匯款」匯出薪轉檔；考勤儀表板**不再**審核薪資          |

## 4. 功能項目一覽

下面每一項都是計劃要有的能力；括號內是白話說明。

### 4.1 個人總覽（首屏）

- **本月摘要卡片** — 出勤天數、請假時數、預估／已核薪資區間（例：115/06）
- **待辦提醒** — 尚有未補卡、假勤待審、薪資審核未送出、主管退回需修改
- **快速入口** — 出勤明細、假別紀錄、薪資明細、本週期審核送出



### 4.2 出勤狀況（唯讀 + 申訴／補說明）

- **月曆／列表** — 每日上下班時間、案場、遲到／早退標記（資料來源：Checkin `打卡紀錄`）
- **異常標示** — 缺卡、僅單邊卡、與排班不符（對照 `排班歷史紀錄`）
- **出勤申訴（Phase A1）** — 逐日明細旁「提出說明」；可附照片／截圖；**15 日內**可送；主管審核通過後可**手動補登**修正紀錄（見 §4.7）
- **匯出本人 PDF／列印** — 選配；優先 Phase 2 再做

#### 4.2.1 技術對接（與管理版共用後端）

| 項目 | 說明 |
| --- | --- |
| 頁面 | `attendance_report.html?mode=personal`（路由 `#/my-personal`） |
| 出勤 API | `get_report`（`userId` 鎖定本人） |
| 假勤 API | `get_my_leave_requests` |
| 遲早退算法 | **後端** `CheckinSystem/Logic_AttendanceReport.js` → `_computeReportLateEarly_` |
| 規格詳述 | `backend/CheckinSystem/SPEC/出勤儀表板與報表規格分析V2.md` §2.5、§2.5.6 |

> **白話**：個人頁的遲到／早退分鐘與管理員出勤儀表板**同一套公式**；後端修正後個人頁自動同步，前端不重算。  
> **彈性早到範例**（表定 08:30、17:30、彈性 30 分）：07:53 到 → 應 17:00 可下班；17:10 走 → 早退 **0** 分。

> 內規提醒：未打卡應當天或隔天補註明原因；不得代打卡（見設計師規範 V7.0）。系統申訴時效為**異常日起 15 日內**（逾期仍可送，標記「逾期申訴」）。



### 4.3 假別紀錄

- **已核准／待審／退回** — 特休、事假、病假、排休等（來源：Checkin `排班歷史紀錄`）
- **剩餘特休** — 依 §4.8 年度統計工作頁計算（已用天數＋人事給額）；無給額時僅顯示「已用 X 天」
- **申請紀錄連結** — 可跳轉既有 Checkin 請假流程（個人頁不重做完整假單 UI 也可）

> 內規提醒：特休原則 3 個工作天前申請；超過 3 天連假需交接。



### 4.4 薪資查看（本人）

- **依發薪週期選擇** — 例 `115/05`、`115/06`（來源：Checkin「薪資發放明細」；送審中見會計主檔「薪資請款申請」）
- **明細列** — 本薪、加班、扣款、實發、備註（來自 `stats_snapshot.payrollBreakdown`／發放快照）
- **狀態** — 草稿／已發放／員工已確認／有異議
- **附件** — 若該期有薪資單 PDF 或截圖，可顯示連結（選配）

**重要**：後端 API 必須帶 `employee_user_id = 登入者`，禁止 list 全表。

### 4.5 每週期「薪資審核」送出（員工 → 會計）

每個發薪週期結束後（或公告的截止日前），員工在**個人出勤頁**完成試算與送審：


| 區塊       | 員工作什麼               | 主管／財務作什麼（會計系統）             |
| -------- | ------------------- | ----------------------------- |
| **薪資確認** | 填加班、遠程津貼、補充說明後送出 | **薪資審核**（≥5）核准／退回；可填獎金、扣款、實發 |
| **假勤** | 系統帶入統計，員工確認        | 對照 Checkin 紀錄（異常另走假勤審核）      |
| **匯款** | —                   | **薪資待匯款**（≥4）匯出中信薪轉檔、寫收支   |


**流程**（2026-07 起）：

```
個人頁送出 → Checkin「薪資核對申請」(已送會計)
  → 會計「薪資請款申請」(待審)
  → payroll_review.html 核准
  → payroll_finance.html 匯出薪轉檔 + 收支登錄
```

- **送出後鎖定** — 員工不可改，除非主管在會計「薪資審核」退回
- **減項金額** — 員工不可改；有異議寫備註，由主管於薪資審核調整後核准
- **考勤審核儀表板** — 已移除「薪資核對」分頁；勿在此審核
- **員工銀行帳戶** — 維護於「員工資料編輯」；會計匯款前檢查 `bank_incomplete`
- **稽核** — approve／reject／export 寫入會計 audit

詳見：`CTBC_SALARY_TRANSFER_SPEC.md`、`accounting-gas/SPEC/PAYROLL_SETTLEMENT_SPEC.md`

### 4.5（舊）每週期「薪資審核」送出（員工 → 主管）— 已廢止

以下為 2026-06 前草案，**主管審核已改至會計**，僅供對照：

| 區塊       | 員工作什麼               | 主管作什麼             |
| -------- | ------------------- | ----------------- |
| **薪資確認** | 勾選「已核對無誤」或勾選有疑問並填說明 | 查看、回覆、必要時修正試算表後重送 |
| **獎金申請** | 填項目（案號、事由、金額、附件）可多筆 | 核准／駁回；核准者寫入薪資或另表  |
| **假勤審核** | 對該週期內異常假勤逐筆確認或補說明   | 對照 Checkin 核准紀錄   |


- **截止時間** — 由主管在後台或試算表設定（例：發薪日前 3 天）
- **送出後鎖定** — 員工不可改，除非主管「退回修改」
- **稽核** — 每次送出、核准、退回寫入 `AUDIT_SPREADSHEET_ID`



### 4.8 個人假勤年度統計（Phase A2）

**目的**：員工在個人頁查看**一整年**（曆年或特休週年，見 §4.8.3）各假別用量與出勤摘要；後端**不每次**從頭掃 `打卡紀錄`＋`排班歷史紀錄`，改讀**獨立工作頁**快取。

#### 4.8.1 畫面（個人頁新分頁「年度統計」）

- **年度選擇** — 僅顯示快取表已有紀錄的曆年（不主動回填舊年）；預設當年或最近有資料之年
- **可見對象** — 個人頁「年度統計」分頁僅 **權限 4 或 5**（主管／人事）可見；一般員工不顯示分頁；後端 `mine`／`refresh` 同門檻
- **摘要列** — 特休／病假／事假／補休／加班（天）、實際出勤、缺勤、遲到／早退（分）
- **特休區塊** — **已用天數**（小數 1 位）；有給額時顯示 **剩餘 = 給額 − 已用**；無到職日僅顯示已用
- **特休使用明細** — 每筆已核准特休可追溯（日期、天數、狀態、假單／班表來源）；可連結「假勤紀錄」分頁
- **各月小計** — 按月折疊：各月特休／病假／事假／補休／加班／出勤／遲早退
- **年底處置（唯讀）** — 顯示人事登錄之未休特休處置（保留／折現）；未登錄時顯示內規說明
- **資料新鮮度** — 顯示 `lastUpdatedAt`；超過 24 小時標「建議重新整理」
- **提示** — 特休建議 3 工作天前申請（內規 V7.0，本模組不硬擋）

#### 4.8.3 特休給額與已用

**已用天數** — 沿用 `Logic_AttendanceReport`（與薪資核對同源）：

| 情境 | 計入天數 |
| --- | --- |
| 全天特休、無打卡 | **1 天** |
| 時段特休（例 `特休[08:00-12:00]`） | 時數 ÷ **8**（例 4h → **0.5 天**） |
| 同日多段特休 | 各段相加 |
| 國定假日上的核准特休 | **計入**（與出勤報表一致） |
| 待審假單 | **不計** |
| 小數顯示 | 前端 **1 位小數** |

**給額（曆年制）** — `_computeAnnualLeaveQuotaForCalendarYear_`：

| 優先序 | 來源 |
| --- | --- |
| 1 | 員工主檔 `特休給額覆寫`（`annualLeaveQuotaOverride`） |
| 2 | 自動：到職日＋勞基法第38條級距表；中途到職依當年在職天數比例；年資跨級距分段 pro-rata |
| — | **廠商**身份：給額 **0** |
| — | **無到職日**：給額空白，個人頁僅顯示已用 |

**剩餘特休** = `annualLeaveQuota` − `annualLeaveDays`（有給額時）。

**統計年度邊界（已決）**：

- **個人頁年度統計**：**曆年制** 1/1～12/31
- **離職結算**：**不在本模組自動處理**；由人事依**週年制**人工結算，與曆年個人頁統計分開

#### 4.8.3d 年底特休折現 — 日薪公式（已鎖定）

**調查結論（2026-07-23）**：會計 `PAYROLL_SETTLEMENT_SPEC` **無**獨立特休折現條文；公司既有薪資試算（`Logic_PayrollReview.js`）已採**勞基法常見實務**，本模組**沿用同一套**，不再另訂公式。

| 項目 | 公式 | 程式對照 |
| --- | --- | --- |
| **月薪員工日薪** | 底薪 ÷ **30** | `PAYROLL_MONTHLY_DAYS_`（=30）；`_payrollHourlyWage_` |
| **日薪員工日薪** | 底薪欄位（即當日薪） | `payType === 'daily'` → `baseSalary` |
| **未休天數** | `max(0, annualLeaveRemaining)`；人事有登錄 `yearEndDispositionDays` 時優先 | API `unusedAnnualLeaveDays` |
| **折現金額試算** | `round(日薪 × 天數)` | API `cashoutAmountPreview`／已登錄折現時 `cashoutAmount` |
| **保留天數** | 人事登錄 `yearEndDispositionDays`（處置＝保留時） | API `carryoverDays` |

**API 處置代碼對照**（試算表 → 前端）：

| 試算表 `yearEndDisposition` | API `disposition` |
| --- | --- |
| `carry_over` | `carryover` |
| `salary` | `cashout` |
| （空） | `null`（僅顯示試算與內規說明） |

**待主管確認**：實際入帳金額以人事年底登錄與次年 1 月薪資發放為準；員工無底薪／薪資類型未設定時，**不顯示**折現試算金額（`cashoutAmountPreview=null`）。

**白話**：月薪的人，一天工資＝月薪除以 30；本來就拿日薪的人，一天就是底薪欄位那個數字。年底沒休完的特休，若要折成現金，就是「一天工資 × 沒休天數」，四捨五入到整數元；畫面上標「試算」，真正發多少還是人事說了算。

#### 4.8.3b 特休明細可追溯

每一筆已核准特休須可對應假單（`排班歷史紀錄`）或班表紀錄；存於 `annualLeaveDetailsJson`，個人頁「特休使用明細」列表顯示，並可前往 §4.3 假勤紀錄查完整申請。

#### 4.8.3c 特休能算出幾天嗎？— **可以**（舊節標題保留供搜尋）

#### 4.8.4 何時重算（避免全表掃描）

| 時機 | 作法 |
| --- | --- |
| **每日排程**（06:00） | ✅ 當年度全員重算（`nightly`） |
| **假勤核准／銷假** | ✅ 事件觸發：核准後 `_triggerLeaveYearStatsRecalcForLeaveDates_`（`leave_approved`） |
| **補登修正打卡** | 同上（若影響缺勤／出勤天數） |
| **個人頁「重新整理統計」** | `refresh`（限流約 5 分鐘 1 次）；缺列首次載入用 `on_demand` |
| **管理端** | perm **> 4（≥5）** 可強制全員重算（寫入 `recomputeTrigger=manual`）；`list` 同門檻 |

重算時仍呼叫 `_getAttendanceDataForReport_(yearStart, yearEnd, userId)` **一次**，結果寫入工作頁；之後查詢只讀一列。

#### 4.8.5 API（草案）

**單一 action**：`leave_year_stats`（比照 `attendance_appeal` 集中一支）

| `mode` | 誰 | 說明 |
| --- | --- | --- |
| `mine` | **perm 4 或 5** | 回傳 `availableYears`（快取已有年度）、`row`（含明細／各月／年底處置）；一般員工（perm 1～3）與最高管理者（perm 6+）不可呼叫 |

**`list` 呼叫例**：`page=attendance_api&action=leave_year_stats&mode=list&year=2026&operatorId=…`  
成功：`{ success:true, data:{ year, lastUpdatedAt, rows:[…] } }`；權限不足：`{ success:false, message:'權限不足…' }`。

#### 4.7.6 API — 集中一支（Checkin `attendance_api`）

**原則**：比照 `get_my_leave_requests`（`mode=self|management`），**只新增一個 action 名稱**，避免 WebApp 白名單與 handler 一直膨脹。

##### 唯一 action：`attendance_appeal`

後端實作：`_handleAttendanceAppeal_(params, method)` 一個入口，內部分流。

**GET**（查詢）


| `mode`    | 誰            | 回傳                                      |
| --------- | ------------ | --------------------------------------- |
| `mine`    | 任何在職員工       | 本人申訴列表（`operatorId` 鎖本人）                |

**POST**（寫入，JSON body）


| `op`      | 誰            | 說明                                                                        |
| --------- | ------------ | ------------------------------------------------------------------------- |
| `submit`  | 員工           | 新增申訴；`operatorId`＝本人；含 `reason`、附件（Base64 或 Drive URL）                    |
| `resolve` | **perm > 3** | `appealId` + `decision`（`approve_fix`／`note_only`／`reject`）+ `reviewNote` |


附件上傳邏輯**內嵌在** `_handleAttendanceAppeal_`，複用假勤 `_handleLeaveRequest_` 的 Drive 上傳，不另開 action。

##### 與既有 API 的關係


| 既有                      | A1 不重複做          |

##### WebApp 註冊（僅 +1 行）

```text

// doPost 路由內：action === 'attendance_appeal' → 同上，method POST
```

**加項／減項試算（B1 UI）— 以勞基法常見換算為準**

| 類別 | 項目 | 試算公式 |
|------|------|----------|
| **加項** | 本薪（月薪） | 底薪 −（底薪÷30×缺勤日）−（底薪÷30×事假日）−（底薪÷30×病假日×0.5）；缺勤取自 `stats.absent`、事假取自 `stats.personalLeave`、病假取自 `stats.sickLeave`（含半天） |
| **加項** | 本薪（日薪） | 日薪×計薪天數（`daysWorked`＝打卡天＋特休＋補休＋病假×0.5）；事假另列減項（日薪×事假天） |
| **加項** | 全勤獎金 | 全勤上限 −（全勤÷30×缺勤日）；**僅缺勤扣全勤日**；事假／特休／病假／補休不另扣全勤（優於勞基法病假全勤比例扣發）；遲早退改依時間扣款 |
| **加項** | 交通／其他津貼、遠程津貼 | 員工設定＋員工自填遠程 |
| **加項** | 加班費 | 勞基法第24條平日延長：時薪=底薪÷30÷8；前2h×4/3、次2h×5/3、其餘×5/3 |
| **減項** | 事假扣款（日薪） | 日薪×`stats.personalLeave`；月薪已併入本薪試算，不另列 |
| **減項** | 病假半薪（月薪） | 已併入本薪試算：底薪÷30×`stats.sickLeave`×0.5（勞基法第43條）；日薪則以計薪天數內病假×0.5 給半日薪，不另列 |
| **減項** | 遲到早退扣款 | 時薪×（`lateMinutesPayroll`+`earlyMinutesPayroll`）÷60；分鐘數**已扣除班表彈性30分** |
| **減項** | 僅單次打卡日遲早 | **暫不計入試算**；畫面仍顯示總遲早分鐘，該日分鐘記入 `lateMinutesHeldSinglePunch`／`earlyMinutesHeldSinglePunch`；員工應於出勤頁**申訴**（類型「僅單次打卡」），主管審核調整後再結算 |
| **減項** | 勞健保自付 | 依投保級距＋眷屬人數 |
| **減項** | 勞退自提 | 依投保級距 × 自提比例（0～6%，員工資料欄位） |
| **減項** | 其他扣款 | 主管審核時手動填入 |

| GET `history`／`payslip` | 員工 | 已發放歷史與明細 |

| POST `resolve` | — | 已改由會計 `payroll_request_approve`／`reject` |

試算表分頁：`薪資核對申請`。`statsSnapshot.payrollBreakdown` 存加減項；`employeeSelfReviewed` 區分員工自審／主管代開。

**時機**：會計薪資試算表／發放紀錄與 Checkin 個人頁 API **完全串接後**再實作 UI；本階段僅鎖規格。

**目的**：員工在個人頁除「當期試算核對」外，可查看**已發放**的歷史實領，作為對帳與存查依據（非試算、以主管／會計確認後金額為準）。
