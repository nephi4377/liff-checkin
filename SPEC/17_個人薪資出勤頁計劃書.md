# 個人薪資出勤頁 — 計劃書（草案）

**版本**：v0.2（2026-06-27）  
**狀態**：Phase A0 已上線；Phase A1 出勤申訴已決策，待開發  
**關聯**：`15_會計系統模組規格書.md` Phase 2e 薪資、`CheckinSystem` 打卡／假勤

---

## 1. 這是什麼？

給**每一位員工**用的「個人專區」：登入後只看**自己的**出勤、假別、薪資，不用進會計後台，也看不到別人的資料。

開發期先限**權限 ≥ 5**（主管／老闆）試用；確認流程沒問題後，再開放給**一般在職員工**（仍只能看本人）。

---

## 2. 為什麼要做？

| 現況 | 問題 |
|------|------|
| 薪資在試算表，只有權限 ≥ 4 的財務／主管能看 | 員工不知道自己當月怎麼算、有疑問只能口頭問 |
| 出勤、請假在 Checkin 系統 | 和薪資分開，對帳時容易漏看 |
| 獎金、補卡、假勤異常 | 多半 LINE 或口頭，缺少「每個發薪週期一次交齊」的入口 |

**目標**：一個頁面把「我這個月上了幾天班、請了什麼假、薪資怎麼來、有沒有要補說明的」集中在一起，並留下紀錄。

---

## 3. 誰能用？

| 階段 | 權限 | 說明 |
|------|------|------|
| **開發試用** | ≥ 5 | 入口先隱藏或灰階，僅內部測流程 |
| **正式上線** | 在職員工（建議 ≥ 1） | 後端強制「只能查自己的 user_id」，不可改參數偷看他人 |
| **管理端** | ≥ 4（維持現狀） | 會計後台仍可看全員薪資、審核員工送出的說明 |

---

## 4. 功能項目一覽

下面每一項都是計劃要有的能力；括號內是白話說明。

### 4.1 個人總覽（首屏）

- **本月摘要卡片** — 出勤天數、請假時數、預估／已核薪資區間（例：115/06）
- **待辦提醒** — 尚有未補卡、假勤待審、薪資審核未送出、主管退回需修改
- **快速入口** — 出勤明細、假別紀錄、薪資明細、本週期審核送出

### 4.2 出勤狀況（唯讀 + 申訴／補說明）

- **月曆／列表** — 每日上下班時間、案場、遲到標記（資料來源：Checkin `打卡紀錄`）
- **異常標示** — 缺卡、僅單邊卡、與排班不符（對照 `排班歷史紀錄`）
- **出勤申訴（Phase A1）** — 逐日明細旁「提出說明」；可附照片／截圖；**15 日內**可送；主管審核通過後可**手動補登**修正紀錄（見 §4.7）
- **匯出本人 PDF／列印** — 選配；優先 Phase 2 再做

> 內規提醒：未打卡應當天或隔天補註明原因；不得代打卡（見設計師規範 V7.0）。系統申訴時效為**異常日起 15 日內**（逾期仍可送，標記「逾期申訴」）。

### 4.3 假別紀錄

- **已核准／待審／退回** — 特休、事假、病假、排休等（來源：Checkin `排班歷史紀錄`）
- **剩餘特休** — 若 Checkin 或人事表有統計則顯示；沒有則 Phase 2 再補
- **申請紀錄連結** — 可跳轉既有 Checkin 請假流程（個人頁不重做完整假單 UI 也可）

> 內規提醒：特休原則 3 個工作天前申請；超過 3 天連假需交接。

### 4.4 薪資查看（本人）

- **依發薪週期選擇** — 例 `115/05`、`115/06`（來源：會計 `PAYROLL_SPREADSHEET_ID` 薪資分頁）
- **明細列** — 本薪、加班、扣款、實發、備註（欄位對齊 `MasterSchema` payroll entity）
- **狀態** — 草稿／已發放／員工已確認／有異議
- **附件** — 若該期有薪資單 PDF 或截圖，可顯示連結（選配）

**重要**：後端 API 必須帶 `employee_user_id = 登入者`，禁止 list 全表。

### 4.5 每週期「薪資審核」送出（員工 → 主管）

每個發薪週期結束後（或公告的截止日前），員工在個人頁一次完成：

| 區塊 | 員工作什麼 | 主管作什麼 |
|------|------------|------------|
| **薪資確認** | 勾選「已核對無誤」或勾選有疑問並填說明 | 查看、回覆、必要時修正試算表後重送 |
| **獎金申請** | 填項目（案號、事由、金額、附件）可多筆 | 核准／駁回；核准者寫入薪資或另表 |
| **假勤審核** | 對該週期內異常假勤逐筆確認或補說明 | 對照 Checkin 核准紀錄 |

- **截止時間** — 由主管在後台或試算表設定（例：發薪日前 3 天）
- **送出後鎖定** — 員工不可改，除非主管「退回修改」
- **稽核** — 每次送出、核准、退回寫入 `AUDIT_SPREADSHEET_ID`

### 4.6 通知（選配 Phase 1.5）

- LINE 推播：週期開放審核、主管退回、補卡提醒、**申訴待審**
- 可沿用 accounting-gas 或 Checkin 既有 LINE 通道

### 4.7 出勤申訴（Phase A1）— 已決策

**目的**：員工對系統判定的遲到、缺卡、離場未填等提出說明；**審核通過後由主管執行手動補登**（不自動改打卡）。

#### 4.7.1 決策摘要（2026-06-27）

| 項目 | 決策 |
|------|------|
| 修正方式 | **B**：審核通過 → 主管在出勤儀表板**手動補登**（留審核人、申訴編號） |
| 審核者 | **權限 > 3**（即 perm ≥ 4，與假勤審核、出勤儀表板同一條件） |
| 時效 | **異常日起 15 日內**為一般申訴；超過仍可送，狀態加標 **逾期申訴** |
| 附件 | **要** — 照片、截圖（沿用假勤／施工回報類 Drive 上傳模式） |
| API | **不新增多支 action** — 只加 **一支** `attendance_appeal`，用 `mode`／`op` 分流（見 §4.7.6） |

#### 4.7.2 員工端（`#/my-personal` 出勤分頁）

- 有異常標記的日期列顯示 **「提出說明」**
- 表單：日期（鎖定）、異常類型（自動帶入或選）、原因（必填）、附件（選填，可多檔）
- 同一員工、同一日期、同一異常類型：**僅一筆「待審」**（避免重複送）
- 送出後顯示狀態：待審／已備註／已修正／已駁回／逾期申訴
- **已修正**：表示主管已補登並結案

#### 4.7.3 主管端

- 入口：假勤審核儀表板新增分頁 **「出勤申訴」**（不另開新路由亦可）
- 列表：**perm > 3** 可看全員待審（與假勤審核同一批人）
- 每筆可：**核准（需補登）**／**僅備註不修正**／**駁回**（必填回覆）
- **核准（需補登）**：開啟既有補登流程（帶入申訴的日期、建議上下班時間）；補登成功後申訴狀態 → **已修正**，寫入 `manualCheckinRef`
- 所有操作寫入稽核（誰、何時、原狀態→新狀態）

#### 4.7.4 狀態機

```
待審 ──核准(需補登)──▶ 待補登 ──主管補登成功──▶ 已修正
  │                      │
  ├──僅備註──────────────▶ 已備註
  └──駁回──────────────▶ 已駁回（員工可改內容再送一筆新編號）
```

#### 4.7.5 試算表 `出勤申訴`（Checkin 試算表新分頁）

| 欄位 | 說明 |
|------|------|
| appealId | 申訴編號（時間戳或 UUID） |
| submitTime | 送出時間 |
| employeeUserId / employeeName | 申請人 |
| employeeGroup | 申請人組別（`group`，如台南店） |
| targetDate | 異常日期 YYYY-MM-DD |
| anomalyType | 遲到／缺卡／僅單次打卡／離場未填／其他 |
| systemSnapshot | 送出當下系統顯示（JSON 或文字：上下班、遲到分、狀態） |
| reason | 員工說明 |
| attachmentUrls | 附件連結（Drive，`;` 分隔） |
| isLateSubmit | 是否逾期（>15 日） |
| status | 待審／待補登／已備註／已修正／已駁回 |
| reviewerUserId / reviewerName | 審核人 |
| reviewTime / reviewNote | 審核時間／回覆 |
| manualCheckinRef | 關聯補登紀錄（日期時間＋審核人） |

#### 4.7.6 API — 集中一支（Checkin `attendance_api`）

**原則**：比照 `get_my_leave_requests`（`mode=self|management`），**只新增一個 action 名稱**，避免 WebApp 白名單與 handler 一直膨脹。

##### 唯一 action：`attendance_appeal`

後端實作：`_handleAttendanceAppeal_(params, method)` 一個入口，內部分流。

**GET**（查詢）

| `mode` | 誰 | 回傳 |
|--------|-----|------|
| `mine` | 任何在職員工 | 本人申訴列表（`operatorId` 鎖本人） |
| `pending` | **perm > 3** | 待審／待補登列表（全員，不分組別） |
| `status` | 員工 | 可選：帶 `targetDate` 查該日是否已有申訴（個人頁列上顯示狀態用） |

共用參數：`operatorId`（必填）、`page=attendance_api`、`action=attendance_appeal`

**POST**（寫入，JSON body）

| `op` | 誰 | 說明 |
|------|-----|------|
| `submit` | 員工 | 新增申訴；`operatorId`＝本人；含 `reason`、附件（Base64 或 Drive URL） |
| `resolve` | **perm > 3** | `appealId` + `decision`（`approve_fix`／`note_only`／`reject`）+ `reviewNote` |

附件上傳邏輯**內嵌在** `_handleAttendanceAppeal_`，複用假勤 `_handleLeaveRequest_` 的 Drive 上傳，不另開 action。

##### 與既有 API 的關係

| 既有 | A1 不重複做 |
|------|-------------|
| `get_report` | 仍負責出勤明細；申訴不改報表算法 |
| `add_manual_checkin` | 審核通過後仍呼叫此支補登 |
| `get_my_leave_requests` | 假勤走假勤；申訴只做出勤異常 |

##### WebApp 註冊（僅 +1 行）

```text
'attendance_appeal': () => _handleAttendanceAppeal_(params, 'GET')   // doGet
// doPost 路由內：action === 'attendance_appeal' → 同上，method POST
```

#### 4.7.7 驗收

- [ ] 員工只能送、只能看自己的申訴
- [ ] 15 日內／逾期標記正確
- [ ] 附件可上傳、主管可開啟
- [ ] 審核通過 + 補登後，報表該日紀錄更新、申訴為「已修正」
- [ ] perm ≤ 3 無法審核、無法代送

## 6. 建議開發分期

### Phase A0 — 個人出勤＋假勤（✅ 已上線 2026-06-27）

- 入口：HUB「我的出勤與假勤」`#/my-personal`
- 個人模式：查本人出勤、假勤分頁；perm &lt; 4 不可進管理版
- 後端：`get_report` 權限鎖定；排班制國定假仍計遲早退
- **驗收**：全員可開個人頁；改 URL 無法看他人

### Phase A1 — 出勤申訴（下一個）

- 試算表 `出勤申訴` + **單一 API** `attendance_appeal`（§4.7.6）
- 個人頁「提出說明」+ 附件
- 假勤審核儀表板加「出勤申訴」分頁（**perm > 3**）+ 串 `add_manual_checkin`
- **驗收**：見 §4.7.7

### Phase B — 週期審核送出（薪資）

- 新建試算表分頁：`薪資審核`、`獎金申請`
- 員工送出、主管在會計後台或個人頁（≥4）審核
- 狀態機：草稿 → 已送出 → 已確認／退回

### Phase C — 薪資個人頁 + 通知

- 個人頁加「薪資」分頁
- LINE 提醒（申訴、週期審核、補卡）

## 8. 權限與資安

- 所有 `personal_*` action：`auth.user_id` 必須等於查詢對象
- 薪資、獎金 API：`appendAudit` 記錄 `view_own_payroll`、`submit_payroll_review`
- 前端不 cache 薪資到 localStorage（或 TTL 極短、僅 sessionStorage）
- 涉及個資：**待主管確認**保存年限、離職後是否關閉入口

**風險提醒**：此案涉及薪資個資與勞動紀錄，建議主管確認流程與法規後再正式對全員開放。
