# 會計與主檔快取策略

**版本**：v1.0（2026-06-20）  
**關聯**：[`15_會計系統模組規格書.md`](15_會計系統模組規格書.md)、[`17_個人薪資出勤頁計劃書.md`](17_個人薪資出勤頁計劃書.md)、backend `accounting-gas/SPEC/ACCOUNTING_MASTER_DATA_SPEC.md`

---

## 1. 快取原則（定案）

使用者少、主檔不常變，採 **「TTL 盡量長 + 有改動立刻更新 + 有機會就背景重讀」**。

| 原則 | 白話 |
|------|------|
| **長 TTL** | 廠商、案場、會計 bootstrap 等不常變的資料，前端可存 **3 天**；後端 GAS `ScriptCache` 最長 **6 小時**（平台上限 21600 秒） |
| **寫入即失效** | CRUD 成功後：前端 `patch*` 局部更新；後端清記憶體索引或 `cache.remove` |
| **Stale-While-Revalidate（SWR）** | 有快取先顯示，同時背景打 API 重讀，成功後靜默覆寫 |
| **不 cache 敏感資料** | 薪資、個人出勤（規劃中）預設不進 `localStorage`，或僅 `sessionStorage` 極短 TTL |
| **GAS 與前端分工** | 6 小時以上的「長快取」放瀏覽器；GAS 只 cache 減少跨試算表讀取的高成本路徑 |

**決策背景（2026-06-20）**

- 案場 map、員工權限：GAS 拉滿 **6 小時**
- 廠商、案場（HUB／會計表單）：**3 天**
- 會計 bootstrap：**3 天**（≥ 使用者要求的 24 小時）
- HUB 當月班表：維持 **7 天**（與案場／廠商分開，變動頻率不同）

## 2. 儲存層級

```
┌─────────────────────────────────────────────────────────┐
│  瀏覽器 sessionStorage   ← 會計 bootstrap（同分頁、依 userId）│
│  瀏覽器 localStorage     ← HUB 員工／案場／班表              │
├─────────────────────────────────────────────────────────┤
│  GAS ScriptCache         ← 員工權限、案場 map、LINE 暫存…   │
│  GAS 函式記憶體 _cache    ← 單次 execution 內訂編案號索引     │
├─────────────────────────────────────────────────────────┤
│  Google Sheets           ← 真實資料來源                     │
└─────────────────────────────────────────────────────────┘
```

## 3. accounting-gas（後端 GAS）快取一覽

| # | 用途 | Cache Key | TTL | 常數／檔案 | 失效／更新 |
|---|------|-----------|-----|------------|------------|
| 1 | **員工權限**（LIFF 驗證用） | `acct_emp_{userId}` | **6 小時** | `AUTH_EMPLOYEE_CACHE_TTL_SEC_` ← `GAS_SCRIPT_CACHE_MAX_SEC_`（21600）<br>`core/AuthBridge.js` | `invalidateAuthEmployeeCache_(userId)`（已實作，待接 CRUD 觸發） |
| 2 | **案場 map**（毛利模組讀 Checkin「案場資料」） | `margin_site_map_v1` | **6 小時** | `MARGIN_SITE_MAP_CACHE_TTL_SEC_`<br>`master/MarginModule.js` | `invalidateMarginSiteMapCache_()`（已實作）；Checkin 改案場**不會**自動清此 key（見 §6） |
| 3 | **訂編案號索引**（記憶體） | `buildOrderProjectMetaIndex_._cache` | **單次 execution** | `master/MarginModule.js` | `order_project_map` CRUD 後 `afterSheetCrudMutate_()` 清 null |
| 4 | LINE 待傳圖片佇列 | `ACCT_IMG_dm_{userId}` | **120 秒**（put 150 秒） | `ACCOUNTING_IMG_TTL_SEC_AG_`<br>`AccountingLineIngest.js` | 送出 ingest 或逾時 |
| 5 | LINE 待解析文字 | `ACCT_ING_dm_{userId}` | **120 秒**（put 150 秒） | `ACCOUNTING_PENDING_TEXT_TTL_SEC_AG_` | 完成 ingest 時 `remove` |
| 6 | LIFF 相片 token | `ACCT_LIFF_T_{token}` | **600 秒** | `ACCOUNTING_LIFF_PHOTO_TOKEN_TTL_SEC_AG_` | 一次性／逾時 |

**全域常數**

| 常數 | 值 | 檔案 |
|------|-----|------|
| `GAS_SCRIPT_CACHE_MAX_SEC_` | 21600（6 小時） | `config_.js` |

**備註**：`accounting_bootstrap` **沒有** GAS 端 ScriptCache，每次 API 都讀試算表；長 TTL 在前端 `AccountingCache`。

## 6. 失效與 SWR 流程摘要

### 6.1 會計主檔（廠商等）

```
使用者開 index.html
  → AccountingCache.load
      ├─ sessionStorage 未過期 → 立刻顯示 + 背景 accounting_bootstrap
      └─ 無快取／過期 → await bootstrap → write sessionStorage

vendors.html 新增／修改廠商
  → GAS crud_* 成功
  → AccountingCache.patchVendor（前端即時）
  → 不需清 TTL；下次 load 背景仍會對齊雲端
```

### 6.2 訂編案號對照

```
GAS crud order_project_map
  → afterSheetCrudMutate_()
  → buildOrderProjectMetaIndex_._cache = null
```

### 6.3 案場（跨系統注意）

| 系統 | 案場快取 | Checkin 改案場後 |
|------|----------|------------------|
| Checkin | `SITES_DATA_CACHE_V3` | 立即 remove |
| accounting-gas | `margin_site_map_v1` | **不會**自動 remove；最慢 **6 小時** TTL 過期 |
| CODING HUB | `spa_hub_projects` | 下次開 HUB 背景 API 更新（SWR） |

若需會計端案場「改完馬上對」，可後續加：Checkin 寫案場時呼叫 accounting-gas 清 cache action，或縮短 #2 TTL。

## 7. 已知限制

1. **GAS ScriptCache 上限 6 小時**，無法在後端做 3 天 TTL；長快取必須放前端。
2. **ScriptCache 單 key 最大 100KB**；大量 JSON 需 gzip 或拆 key（Checkin 出勤 gzip 快取見 Checkin SPEC §6.1）。
3. **`ACCOUNTING_MASTER_DATA_SPEC.md`**（backend）內 mermaid 若仍寫 5 分鐘，以**本文件**為準。
4. **員工權限**變更後，accounting `acct_emp_*` 最慢 6 小時才過期；可手動呼叫 `invalidateAuthEmployeeCache_` 或等 TTL。
