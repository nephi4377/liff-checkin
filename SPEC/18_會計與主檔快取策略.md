## 1. 快取原則（定案）

使用者少、主檔不常變，採 **「TTL 盡量長 + 自己改過就立刻改快取那一筆 + 有快取時 24 小時 SWR」**。

| 原則 | 白話 |
|------|------|
| **長 TTL** | 廠商、案場、會計主檔包等不常變的資料，前端可存 **3 天**；後端 GAS 暫存最長 **6 小時**（平台上限） |
| **寫入即更新** | 新增／修改主檔成功後，**優先就地改快取裡那一筆**（patch）；改不到才整包清掉，下次再完整重抓 |
| **讀取 SWR** | 有快取且未過 3 天 → **立刻顯示**；若距上次抓取 **> 24 小時** → **背景**再打 `accounting_bootstrap`（不擋畫面） |
| **背景重抓（SWR）** | 會計 bootstrap、HUB 員工／案場／班表、會計列表 — bootstrap／列表均 **24 小時** SWR（單人模式） |
| **不 cache 敏感資料** | 薪資、個人出勤（規劃中）預設不進 `localStorage`，或僅 `sessionStorage` 極短 TTL |
| **GAS 與前端分工** | 6 小時以上的「長快取」放瀏覽器；GAS 只暫存「跨試算表讀取很貴」的路徑（員工權限、案場 map 等） |

**決策背景（2026-06-20）**

- 案場 map、員工權限：GAS 拉滿 **6 小時**
- 廠商、案場（HUB／會計表單）：**3 天**
- 會計 bootstrap：**3 天**（≥ 使用者要求的 24 小時）
- HUB 當月班表：維持 **7 天**（與案場／廠商分開，變動頻率不同）

**身分與 operator 傳承**（快取以外）：見 [`19_HUB與會計全域身分傳承.md`](19_HUB與會計全域身分傳承.md)。

## 2. 儲存層級

```
┌─────────────────────────────────────────────────────────┐
│  瀏覽器 sessionStorage   ← 會計 bootstrap（同分頁、依 userId）│
│  瀏覽器 localStorage     ← HUB 員工／案場／班表              │
├─────────────────────────────────────────────────────────┤
│  GAS ScriptCache         ← 員工權限、案場 map、LINE 暫存…   │
│  GAS 函式記憶體 _cache    ← 單次 execution 內訂編案號索引     │
├─────────────────────────────────────────────────────────┤
│  Google Sheets           ← 真實資料來源                     │
└─────────────────────────────────────────────────────────┘
```

| 項目 | 值 |
|------|-----|
| **模組** | `AccountingCache`（`shared/js/accounting_cache.js`） |
| **儲存** | `sessionStorage` |

## 6. 失效與讀取流程摘要

### 6.1 會計主檔（bootstrap）

```
使用者開會計頁
  → 讀主檔快取
      ├─ 未過 3 天 → 立刻顯示
      │     └─ 距上次抓取 > 24h → 背景再打 accounting_bootstrap（SWR，不擋畫面）
      └─ 沒有／過期／曾整包清除／force → 同步向後端要一整包 → 寫進分頁暫存

廠商名冊新增／修改廠商
  → 後端存檔成功並回傳該筆資料
  → 前端就地改快取裡那一筆（patch）
  → 同頁列表直接用更新後的快取重畫（不必整包重抓）
  → 若回傳缺編號、無法 patch → 整包清掉；下次進頁再完整重抓
```

**實作責任**

| 層 | 誰負責 |
|----|--------|
| 前端 | 新增／修改主檔成功 → `AccountingCache.afterCrudSuccess`（先 patch，失敗才 clear） |
| 前端 | 建立廠商資料夾、上傳廠商照片成功 → 視同廠商主檔變更，同樣走 afterCrudSuccess／欄位 patch |
| 前端 | 頁面可呼叫 `patchMaster`／`patchVendorFields`；**勿**自己另寫一套快取格式 |
| GAS | 訂編案號等變更後清「單次執行內」的索引記憶體 |

### 6.1.1 CRUD → 快取更新對照（定案）

| entity／動作 | 前端主檔快取怎麼處理 | GAS 快取 | 下次讀取 |
|--------------|----------------------|----------|----------|
| `vendor` | **優先 patch 該筆**；失敗才整包清除 | — | 有快取就直接用；被清掉才 bootstrap |
| `vendor_line_binding` | 同上 | — | 同上 |
| `payee` | 同上 | — | 同上 |
| `category` | 同上 | — | 同上 |
| `order_project_map` | 同上 | 清單次執行索引 | 同上 |
| 建立廠商資料夾／上傳廠商照片 | patch 廠商資料夾欄位（或走 afterCrudSuccess） | — | 同上 |
| 帳本狀態／廠商付款／毛利等交易 | **不動**主檔包 | 依模組（見 §3） | 各頁即時查 |
| Checkin 員工／案場 CRUD | HUB 員工／案場快取（HUB 端） | Checkin 暫存 | HUB 背景更新；會計表單可補抓 |

**原則**：會寫進主檔包的資料，CRUD 成功後**優先就地改那一筆**（省一次整包重抓）；只有改不到時才整包清除。  
**限制**：這只保證「**自己這台瀏覽器、自己改過的**」是新的；別人在別台改的，最快要等 3 天過期或手動強制重載才看得到。

### 6.2 訂編案號對照

```
GAS crud order_project_map
  → afterSheetCrudMutate_()
  → buildOrderProjectMetaIndex_._cache = null
```

### 6.3 案場（跨系統注意）

| 系統 | 案場快取 | Checkin 改案場後 |
|------|----------|------------------|
| Checkin | `SITES_DATA_CACHE_V3` | 立即 remove |
| accounting-gas | `margin_site_map_v1` | **POST** `invalidate_margin_site_map`（需 Checkin Properties 有 `ACCOUNTING_GAS_WEB_APP_URL`＋與會計相同的 `INGEST_SECRET`）；失敗時仍最慢 **6 小時** TTL |
| CODING HUB | `spa_hub_projects` | 下次開 HUB 背景 API 更新（SWR） |

實作：`CheckinSystem/SiteLogic.js` → `_notifyAccountingMarginSiteMapInvalidate_()`；`accounting-gas/WebApp.js` action `invalidate_margin_site_map`。

## 7. 已知限制

1. **GAS ScriptCache 上限 6 小時**，無法在後端做 3 天 TTL；長快取必須放前端。
2. **ScriptCache 單 key 最大 100KB**；大量 JSON 需 gzip 或拆 key（Checkin 出勤 gzip 快取見 Checkin SPEC §6.1）。
3. **`ACCOUNTING_MASTER_DATA_SPEC.md`**（backend）內 mermaid 若仍寫 5 分鐘，以**本文件**為準。
4. **員工權限**變更後，accounting `acct_emp_*` 最慢 6 小時才過期；可手動呼叫 `invalidateAuthEmployeeCache_` 或等 TTL。
