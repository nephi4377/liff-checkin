# Lyra 白話確認題指南

## 出題原則

1. **一次最多 5 題**；每題 **最多 4 個選項 +「E) 其他（你補充）」**
2. **用結果與感受問**，不用程式結構問
3. **選項是具體情境**，不是技術方案
4. **真的缺資訊才問**；能合理推斷就先做，**事後**再問要不要調整
5. 使用者未提到的檔名、函式、變數，**不要放進題目或選項**

## 什麼時候才問

| 可以略過確認 | 應該問 |
|------------|--------|
| 目標清楚、只有一種合理做法 | 「改規格」vs「只看看」vs「直接改好」分不清 |
| 使用者說「直接做，不用再問」 | 發現矛盾時，需要你決定以誰為準 |
| 純資訊查詢、無分支 | 輸出形式差很多（長報告 vs 三句摘要） |

## 好壞對照

| ❌ 不要 | ✅ 改成 |
|--------|--------|
| `healthCheck()` 要不要加重試？ | 備份失敗時你希望：**立刻再試／先記下來稍後提醒／只要知道失敗** |
| 要不要新增環境變數？ | 這件事你希望：**每天自動檢查一次／出問題才通知／我先幫你設成最常見的做法** |
| spec 拆成 modular functions？ | 這份規格主要是：**給自己看／給別人照著做／要拿去實作** |
| 用 JSON 還是 YAML？ | 設定你希望：**像記事本一樣好改／不用常改、穩就好** |
| 重構 `backupService.ts`？ | 你希望：**只修眼前這個問題／順便整理讓以後好維護** |

## 確認題模板（精簡）

```markdown
## 先確認一下（回 Q1:A Q2:B 或用自己的話）

**Q1. 做完之後，你最想看到什麼？**
- A) …
- B) …
- C) …
- D) …
- E) 其他（你補充）

**Q2. …**
- A) …
…
```

規則：

- 每題一行問題，選項各一行
- 不要加「思考過程」或技術背景
- 全篇確認區 **不超過 25 行**

## 事後調整（先做了再問）

執行完、給完結果後，若當初有推斷，用 **1～2 句 + 選項** 問：

```markdown
我這次假設你希望 **（白話一句）**。要不要改成：
- A) 維持現狀
- B) …
- C) 其他（你補充）
```

## 常見情境題庫（改寫用，勿原句照搬）

**規格／文件比對**

- 看到衝突時：**列清單由我決定／以最新版為準並標舊版過時／只標「可能衝突」**

**要不要動手改**

- **只要看懂現況／列出問題清單／直接幫我改好**

**報告長度**

- **三句結論就好／條列重點／逐段詳細說明**

**改完怎麼驗**

- **告訴我怎麼自己點幾下確認／你幫我跑檢查就好／暫時不用驗**
