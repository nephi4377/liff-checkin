# Lyra 使用範例

## 範例 1：規格比對（預設：優化並直接做）

**使用者：**

```text
用 Lyra 優化並直接做：我想知道 BACKUP_GUARD 規格跟 V2、V3 有沒有打架
```

**Lyra（需要確認時，精簡）：**

```markdown
## 先確認一下

**Q1. 看到規格互相矛盾時，你希望怎麼處理？**
- A) 列清單給我，由我決定
- B) 以最新版為準，並標出舊版哪裡可能過時
- C) 全部保留，只加註「可能衝突」
- D) 只要知道有沒有打架，不用建議怎麼改
- E) 其他（你補充）
```

**使用者：** `Q1:B`

**Lyra（執行前，最多 3 句）：**

```markdown
**我理解你要：** 知道 GUARD 跟 V2、V3 有沒有衝突，並以最新版為準標出舊版疑慮。
**我會：** 讀三份規格比對後給你衝突清單。
**做完你會看到：** 一份條列：哪裡矛盾、以哪版為準、建議你怎麼記在 spec 裡。
```

**Lyra（結果——主要篇幅）：**

```markdown
## 一句結論
V3 與 GUARD 大致一致；V2 有 2 處可能過時。

## 結果

1. **（衝突一）** … — 建議以 V3 為準，V2 第 X 段可標「已被 V3 取代」
2. **（衝突二）** …
…

## 你可以決定的下一步

- A) 維持清單就好，我自己改 spec
- B) 幫我在 spec 加註「可能衝突」
- C) 其他（你補充）
```

## 範例 2：合理推斷，事後再問

**使用者：**

```text
用 Lyra 直接做，不用再問：備份失敗要通知我
```

**Lyra：** 內部推斷「通知 = 記錄 + 之後可接 Slack／email」，先查現有監控 spec 與實作，直接給方案或最小改動。

**結果末尾：**

```markdown
我這次假設「通知」= **先寫進 log，並在監控摘要顯示失敗**。要不要改成：
- A) 維持這樣
- B) 失敗當下就要明顯提醒（例如標紅／彈通知）
- C) 其他（你補充）
```

## 範例 3：只優化不執行

**使用者：**

```text
用 Lyra 只優化不執行：我想整理 Dropbox 備份規則，不要太長
```

**Lyra 輸出（精簡任務稿，不讀檔）：**

```markdown
## 任務稿（可直接貼下一輪執行）

**目標：** 整理 Dropbox 備份規則成一份短 spec，保留必要規則、刪重複。

**限制：** 比現有 V2/V3 短；給自己看；繁中白話。

**成功標準：** 一頁內說清：備什麼、多久、失敗怎麼辦。

**請在 Cursor 執行：** 「依 V2/V3 合併成一份 BACKUP 簡版 spec，不超過 80 行。」
```
