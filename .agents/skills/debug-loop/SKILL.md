# Debug Loop

## 何時使用
- 使用者明確說「除錯循環」。
- 問題需要可重現測試、根因分析、修正、再驗證。
- 問題有回歸風險，不能只靠猜測修補。

## 目標
用固定流程把問題處理到可驗證結束，不只修表面症狀。

## 流程
1. **理解狀況**
   - 釐清需求、輸入、預期、環境、相依。
   - 資訊不足時先問關鍵問題並暫停，不硬修。

2. **測試（Test）**
   - 至少覆蓋：正常、邊界、異常輸入。
   - 可執行就實跑並引用結果。
   - 無法執行時，明確標記為推論，寫清楚推導過程與失敗點。

3. **分析（Analyze）**
   - 給出根因，不只描述現象。
   - 列出可能副作用與受影響範圍。

4. **修復（Fix）**
   - 先做最小可行修正，再做必要的可讀性與錯誤處理改善。
   - 避免無依據的大改。

5. **再測試（Re-test）**
   - 用與步驟 2 同一組測試驗證修正。
   - 回報是否通過、是否有回歸風險、建議監控點。

## 回覆格式（固定四段）
回覆必須含以下標題，順序建議一致：
- **測試結果**
- **問題診斷**
- **修正後代碼**
- **最終驗證**

## 執行規則
- 不可捏造測試結果、log、檔案內容。
- 若有未確認假設，必須明確標註。
- 若修改規則檔或系統指令，需同步更新當日 `LOG/YYYY-MM-DD_LOG.md`。

## 快速檢查清單
- [ ] 有正常／邊界／異常三類測試
- [ ] 有根因，不是只描述症狀
- [ ] 有可套用修正（diff 或片段）
- [ ] 有再測試與回歸觀察點
