# 新對話交接（Conversation Handoff）

## 何時用

- 使用者要**換新對話**接續同一任務
- 對話太長、要壓縮脈絡給下一個 Agent
- 使用者說「幫我輸出要告知的內容」

## 你要做的事

1. **從本對話整理**（必要）
   - 使用者**最終目標**（一句話 + 子目標）
   - **已完成**（含檔案／規格／部署版號）
   - **未完成／待驗收**
   - **部署狀態**（GAS clasp、CODING upload.bat、是否只在本機）
   - **已知問題／刻意不修**
   - **下一個 Agent 第一件事**

2. **若涉及程式**（建議，有權限就執行）
   - `git status --short`：`CODING/`、`backend/`（或使用者指定的 repo）
   - 最近部署記錄（如 `@74`、deploy 描述）
   - 勿 commit／push／clasp，除非使用者在本輪**另外**明確要求

3. **輸出給使用者**
   - 用下方 **Handoff 模板** 產出**一整塊 markdown**
   - 開頭加一句：「以下可直接複製貼到新對話的第一則訊息」
   - 模板內用**具體路徑、版號、檔名**；避免「如上所述」
   - 繁體中文；技術名詞（action、clasp）可保留英文

4. **不要**
   - 不要把整段對話原文 dump 出來
   - 不要超過 ~80 行（新對話要精簡）
   - 不要擅自執行部署或 upload

## Handoff 模板

產出時**保留標題**，把括號提示換成實際內容：

```markdown
## 接續任務（請先讀完再動手）

### 背景／目標
（一句話總目標；若有子目標列 2～5 點）

### 已完成
- （功能或修復 + 主要檔案路徑）
- （規格／文件若有改也列出）
- **部署**：backend GAS（有／無，版號 @N）；CODING 前端（已 push main／僅本機）

### 待辦（優先序）
1. （最 urgent）
2. …

### 關鍵路徑
| 項目 | 路徑 |
|------|------|
| … | `…` |

### 驗收方式
- （Ctrl+F5、哪一頁、看什麼現象）

### 限制／勿做
- （例：未明說部署不 clasp；SPEC 19 尚未做）
- 回覆用繁體中文、白話

### 建議載入 Skill
- `@GAS-Backend-Expert`（若動 backend）
- `@Cloud-Agent-Runbook`（若本機測前端）
- （其他）

### 請你先
（一句話：例如「執行 CODING upload.bat 上線前端，再協助驗收待付款申請 OCR」）
```

## 品質檢查

輸出前自問：

- [ ] 新 Agent 不看舊對話也能知道**下一步**？
- [ ] 部署狀態是否寫清楚（避免重複 deploy 或以為已上線）？
- [ ] 未 commit 的變更是否對應到具體檔案？
- [ ] 「請你先」是否只有**一個**明確動作？

## 範例觸發

使用者：「我要移到新對話框，幫我輸出要告知的內容」

→ 依本 skill 產出 Handoff 區塊；可順便提醒：「新對話可 @conversation-handoff 請 Agent 更新交接內容」。
