# human comfortable

**管**：真人會看、點、填、讀的介面。  
**不管**：Agent 回覆格式、log／JSON 當預設畫面。

## 先記：人會做不到的事

設計時假設使用者**做不到**下面這些，介面要代勞或降要求：


| 做不到            | 介面怎麼補                    |
| -------------- | ------------------------ |
| 同時想很多事、記不住步驟   | 一屏一事；下一步寫清楚；預設值、自動帶入     |
| 在雜亂畫面找重點       | 留白、主從分明；同區**一個**主按鈕      |
| 讀小字、辨顏色、精準點小目標 | 字夠大；紅綠須配文字；觸控區夠大、間距夠     |
| 懂錯誤碼與術語        | 人話；錯在欄位旁；告訴怎麼修           |
| 猜系統有沒在跑        | 點了立刻有回饋（轉圈、處理中、disabled）；防重按見下節 |
| 承受犯錯、怕搞砸       | 可反悔；危險操作確認且預設取消；少擋全屏     |


## 防重按

對齊上表「猜系統有沒在跑」— 使用者點了，**立刻**要知道「有收到」。


| 情境 | 必做 |
| --- | --- |
| 任何可點動作 | 立刻 `disabled` 或按鈕改「處理中…」；完成／失敗後還原 |
| 送出、核准、儲存、匯出等 | 鎖**觸發的那顆鈕**（或該列／該卡片），**不要**預設鎖整頁所有按鈕 |
| 長操作 | 可先 toast「處理中」，背景繼續；不必卡死整頁等結果 |

**會計模組參考**（`modules/accounting/`）：按鈕包一層「非同步執行」— 自動套上忙碌樣式、擋連點、寫進狀態紀錄。審核頁另有**逐筆佇列**：同一筆在跑只鎖那列按鈕，其他列仍可操作；多筆時底部顯示「處理中 2/5」，取代早年鎖死整頁的做法。

## 三大原則


| 原則          | 必做                                           |
| ----------- | -------------------------------------------- |
| **看哪裡很清楚**  | 標題說這頁在做什麼；摘要優先、細節收起；少用功能藏進階                  |
| **要做什麼很簡單** | 主功能不藏三層；四態（載入／空＋建議／錯＋怎麼修／成功輕提示）；表單有可見標籤、防連點（見 **防重按**） |
| **做錯了不挫折**  | 禁錯誤碼、裸 JSON、只靠顏色、`alert()` 當唯一回饋；刪除／覆蓋要確認或還原 |


**頁面骨架**：`[標題]` → `[可選一行狀態]` → `[一個主按鈕]` → `[摘要／卡片]`

## CODING 網頁（Tailwind）

對齊 `spa/Dashboard.js`；文案 `00-general-plain-language`。spa／modules 適用，WordPress 匯出官網不強制。

淺灰底（`bg-gray-50`）、白卡片（`rounded-xl shadow-sm border-gray-100`）、主色 Blue 600、按鈕 `hover`／`transition`、input `focus:ring-2`、留白 `p-6` 起。

## 單頁殼層（SPA）與版面比例

會計等「外層殼 + 內嵌子頁」模組適用；其他模組若採同一架構也照做。

| 規則 | 必做 |
| --- | --- |
| **狀態紀錄只掛一次** | 狀態欄／操作紀錄掛在**外層殼**；內嵌子頁（`embed=1`）**不要**再掛第二組，也不要因切頁重掛、重播整份紀錄。子頁操作可轉給外層顯示。 |
| **寬螢幕主欄** | 保留左側系統資訊／導覽區；**主內容約佔視窗寬 55–65%**（目標約 60%），右側可有一欄狀態紀錄。 |
| **手機** | 主區改全寬；狀態欄可改底部或收合，勿讓雙欄把內容擠成細條。 |
| **預設畫面** | 狀態紀錄是輔助欄，不是主畫面；主區仍以標題、主按鈕、摘要／表單為主。 |

反例：外層已有「狀態紀錄」，子頁 iframe 再掛一組 → 使用者看到兩欄相同紀錄。
