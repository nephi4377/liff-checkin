# 白話說明

**用白話的方式說明事情，不要用函式；如果要用到函式代表意義，要附上白話說明。**

## 管什麼

- 對使用者**解釋**：發生什麼事、為什麼、接下來會怎樣
- 寫給人看的**摘要、步驟、驗收、變更說明**

**不管**：實際改程式時的識別符號（那時仍照專案慣例寫 code）

## 核心規則

1. **繁體中文**；先結論，再補必要細節
2. **預設不用**函式名、類別名、檔名當主語——改說「這段程式在做什麼」
3. **不得不提**程式識別符號時：`名稱（白話：…）`，白話放括號或下一行
4. **術語**先查 [`SPEC/專有名詞白話對照.md`](../../../SPEC/專有名詞白話對照.md)；表內沒有就**第一次**括號補一句，可順手補表
5. **長度**：簡單問題幾句；對方說「展開／給步驟」再寫長
6. **禁止**整段堆函式名、錯誤碼、JSON、log 當預設說明

## 改寫對照

| 不要這樣說 | 改成白話 |
|------------|----------|
| `fetchOrders()` 回傳空陣列 | 「去資料庫抓訂單」這步沒抓到任何一筆 |
| 在 `Dashboard.js` 改 `renderStats` | 在**儀表板頁面**改「統計數字怎麼顯示」 |
| `catch` 到 `TypeError` | 程式預期是文字，卻拿到空值或格式不對 |
| 呼叫 `gasRun_` 失敗 | 後端小程式沒跑成功（常是權限或試算表連線） |
| diff 在 L42 | 第 42 行附近改了一小段邏輯 |

## 不得不寫程式名時

格式固定：

```text
savePaymentRecord（白話：把這次付款紀錄存進試算表）
```

或：

```text
這次動到的是「儲存付款紀錄」那段程式（程式裡叫 savePaymentRecord）。
```

同一則回覆裡，**同一個名字最多出現一次**；之後一律用白話代稱。

## 回覆形狀（預設）

```markdown
## 一句話
（這次發生什麼／你要知道什麼）

## 白話說明
（用「誰、做什麼、結果」；必要時 1–3 步）

## 若還有程式名
- 名稱（白話：…）
```

對方明確要「給步驟／怎麼驗／先別改」時，可改用 `plain-verification-checker` 或 `noncoder-cursor-orchestrator` 的模板，但**仍遵守本 skill 的用語規則**。

## 好／壞範例

**壞**

> 我在 `processVendorClosure_` 裡把 `normalizeAmount` 的 null check 移到 `validateRow_` 之前，避免 `reduce` 噴錯。

**好**

> 廠商結案時，有一筆金額是空的，加總時整段算不出來。我把「先確認金額有填」移到「開始加總」之前，空值就不會讓整批失敗。  
> （若需對照：`processVendorClosure_`＝廠商結案主流程）

**壞**

> `getDashboardData` timeout 因為 `CacheService` miss。

**好**

> 儀表板載入太久，因為暫存沒命中，每次都重新去後端抓資料。

## 和其他 skill 的分工

| Skill | 分工 |
|-------|------|
| **本 skill** | 用語：白話優先、函式必附解釋 |
| `human-comfortable` | 給**真人點的介面**文案與版面 |
| `plain-verification-checker` | **怎麼自己驗**成功與否 |
| `noncoder-cursor-orchestrator` | 不懂程式時的**小步驟＋先問再做** |

## 觸發句（範例）

- 用白話說明
- 不要講函式
- 聽不懂你在說什麼
- 先解釋，不要改
- 像跟同事講一樣說
